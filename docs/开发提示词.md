## 使用方式

1. 先读 [任务分段书.md](/Users/adlin/Documents/Code/ORIN/docs/任务分段书.md)
2. 从 `P0` 里选择一个还没完成的小任务
3. 单次对话尽量完成一个编号任务
4. 完成后把对应任务从 `[ ]` 改成 `[x]`
5. 如发现任务描述不准，先更新任务书，再继续开发

## 默认工作原则

- 不再默认“页面存在 = 能力完成”
- 不再默认“接口存在 = 已可交付”
- 任何结论都优先以真实代码、真实路由、真实执行结果为准
- 涉及工作流、协作、知识处理时，要同时检查前端、Java 后端、Python AI 引擎三层
- 遇到依赖外部服务的能力，要补“已配置”和“未配置”两种场景

## 每次开始前的检查

开始某个任务前，请先做最少上下文核对：

1. 阅读任务对应模块的当前文档
2. 搜对应前端路由或页面入口
3. 搜对应后端 controller / service
4. 如果涉及执行链，再看 `orin-ai-engine`
5. 看 `git status`，避免覆盖用户已有改动

## 每次结束后的要求

完成任务后请一起做这些事：

1. 更新 [任务分段书.md](/Users/adlin/Documents/Code/ORIN/docs/任务分段书.md) 中对应状态
2. 如果能力边界发生变化，同步更新相关文档
3. 如果补了测试或脚本，说明运行方式
4. 如果还有残余风险，明确写出来，不要用模糊表述带过

## 面向模型的具体指令

如果你是接手开发的模型，请按以下方式工作：

- 优先做“可以验证真闭环”的任务，而不是继续堆入口
- 优先补 smoke、回归、错误语义、配置边界、回滚能力
- 避免把历史文档里的乐观描述继续复制到新文档
- 对发现的伪完成状态，要直接修正文档和状态口径
- 如果一个任务本质上需要拆分，先在任务书里拆开再做
