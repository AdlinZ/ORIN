# ORIN 从当前基线到理想交付的开发规划报告

**文档版本**：v1.0  
**编制日期**：2026-04-11  
**规划周期**：16 周（建议）  
**依据文档**：[系统功能实现评估报告](./系统功能实现评估报告.md)

## 1. 规划目标与成功标准

### 1.1 目标定义
在当前“骨架较完整、闭环成熟度不足”的基础上，用 16 周完成从“可演示/可联调”到“理想交付（成熟平台能力）”的工程化收敛。

### 1.2 最终成功标准（理想交付口径）
满足以下三层标准后，才可对外按“成熟平台”交付：

1. 功能层：四大核心模块（智能体知识与工作流、运行监控、多智能体协作、统一网关与访问控制）均形成稳定闭环并可运维。
2. 集成层：Dify、RAGFlow、AutoGen、CrewAI 完成统一接入模型（Tool / MCP / API）与可观测调用链。
3. 运营层：统一观测、告警、审计、发布、SLA、故障演练体系可执行且有证据。

### 1.3 量化验收 KPI（建议）

- `P0` 阻断故障：连续 14 天为 0（理想交付前最后门槛）
- 核心接口成功率：`>= 99.5%`
- 核心链路 P95 时延：
  - 对话与检索：`<= 2.5s`
  - 工作流执行提交：`<= 1.5s`
  - Trace 查询：`<= 1.0s`
- 关键页面可用性（18 个重点路由）：`>= 98%`
- 核心 smoke 用例通过率：`100%`（后端/前端/AI 引擎）
- 回归自动化通过率：`>= 95%`
- 严重安全问题（高危）：`0`

## 2. 当前基线与差距

### 2.1 当前基线（来自评估报告）

- 骨架完成度：`75% - 80%`
- 真实业务闭环：`50% - 60%`
- 对外稳定交付：`45% - 55%`
- 相对成熟：智能体管理、系统管理
- 关键短板：协作执行链、统一观测口径、多模态外部依赖、外部框架集成可控性、测试与发布基线

### 2.2 到理想交付的核心差距（Gap）

1. 从“有页面/有入口”到“端到端稳定闭环”的差距。  
2. 从“模块可用”到“跨模块统一口径（traceId/taskId）”的差距。  
3. 从“可联调”到“可运维（告警、审计、SLA、演练）”的差距。  
4. 从“局部测试”到“分层质量门禁+持续回归”的差距。

## 3. 总体策略与实施原则

### 3.1 实施策略

1. 先保底后理想：先完成“最低可上线”闭环，再扩到“理想形态”。
2. 双轨并行：
   - 轨道 A：核心闭环与稳定性治理
   - 轨道 B：平台化与运维体系建设
3. 证据驱动验收：每个里程碑必须附“可复现用例 + 日志/截图 + 指标看板”。
4. 统一数据契约优先：前后端字段、错误码、状态机先对齐，再做功能扩展。

### 3.2 交付分层

- L1（可运行）：功能可走通
- L2（可稳定）：异常可处理、可恢复、可观测
- L3（可运维）：可告警、可审计、可回溯、可扩展

## 4. 16 周分阶段开发路线图

## 4.1 阶段划分

| 阶段 | 周期 | 阶段目标 | 里程碑输出 |
| --- | --- | --- | --- |
| Phase 0 | 第 1-2 周 | 基线收敛与统一口径 | 路由/接口/错误码/指标口径基线、验收清单 v1 |
| Phase 1 | 第 3-6 周 | 达成“项目最终最低交付要求” | 核心四模块最小闭环 + 外部集成保底 + 端侧同步最小可用 |
| Phase 2 | 第 7-11 周 | 向“理想交付”扩展核心能力 | 四框架统一接入、双向同步、统一观测关联分析 |
| Phase 3 | 第 12-16 周 | 运维体系与对外交付封板 | SLA、告警演练、性能安全达标、对外能力清单 |

## 4.2 逐阶段详细计划

### Phase 0（第 1-2 周）：基线与契约收敛

**目标**：解决“评估口径不一致、接口字段不一致、观测口径不一致”问题。  
**工作包**：

1. 统一能力状态模型：`可用 / 可联调 / 预留 / 不可用`。
2. 梳理 18 条重点路由，建立“页面-接口-服务-执行器”链路映射表。
3. 发布统一错误码规范（鉴权、配置缺失、外部不可达、执行失败、限流）。
4. 发布统一观测字段规范（成功率、错误率、时延、token、traceId、taskId）。
5. 固化“每模块至少 1 条 smoke”并接入 CI。

**退出标准（DoD）**：

- 路由与接口映射覆盖率 `100%`
- 字段对齐问题清零（至少核心模块）
- CI 可稳定执行 3 条基础 smoke（前端/后端/AI）

---

### Phase 1（第 3-6 周）：最低可上线闭环

**目标**：满足“项目最终最低交付要求（保守可上线）”。  
**工作包**：

1. 运行监控闭环
- `/dashboard/runtime/overview` 与 `/dashboard/runtime/traces` 稳定出数
- Trace 查询闭环：traceId 输入 -> 链路步骤 -> 指标统计
- 观测字段与后端接口一一对齐

2. 协作模块最小闭环
- 任务包：创建 -> 分解 -> 执行 -> 汇总 -> 完成/失败落库
- 串行/并行两种执行模式可复现
- 仪表盘指标可回溯到任务记录

3. 外部集成保底
- Dify、RAGFlow：配置保存 + 连通测试 + 1 次真实调用
- AutoGen、CrewAI：标记为预留并输出明确错误语义

4. MCP 与端侧同步最小可用
- MCP 服务：增删改查 + 测试连接 + 可读失败原因
- Client Sync：查询变更 + 触发同步 + 查看结果
- `/resources/sync` 与 `/control/client-sync` 去重与职责划分

5. 知识库与统计稳定性
- 知识链路 smoke：上传 -> 解析 -> 检索
- 统计页 7 天区间稳定展示

**退出标准（DoD）**：

- “最终最低交付要求”全部打钩
- 连续 7 天无 P0
- 核心演示用例（成功/失败）均可复现并有证据

---

### Phase 2（第 7-11 周）：理想能力扩展

**目标**：向“理想交付要求”推进，补齐平台化短板。  
**工作包**：

1. 外部框架统一接入层
- 建立统一 Adapter 规范（Tool / MCP / API）
- Dify、RAGFlow、AutoGen、CrewAI 接入同一生命周期状态机
- 统一重试、超时、熔断与限流策略

2. 端侧同步增强
- 双向同步
- 断点续传
- Webhook 实时触发
- 冲突检测与冲突解决策略（最后写入/版本优先/人工合并）

3. 统一观测体系成型
- Langfuse + Prometheus/Grafana + 日志聚合整合
- traceId/taskId 全链路关联
- 构建告警规则（失败率、时延、队列堆积、外部依赖失败）

4. 队列与配置中心治理
- 队列优先级、自动重试、死信队列治理
- 配置中心可视化、热更新、版本回滚

**退出标准（DoD）**：

- 四外部框架统一接入可用
- 双向同步可通过全量+增量+冲突场景验收
- 核心告警可触发、可通知、可闭环

---

### Phase 3（第 12-16 周）：封板与对外交付

**目标**：形成“成熟平台”的质量与运营保障。  
**工作包**：

1. 性能与容量
- 并发压测、容量基线、瓶颈修复
- 核心接口 P95/P99 达标

2. 安全与合规
- RBAC 穿透测试
- JWT、限流、审计日志一致性检查
- 敏感信息脱敏与密钥管理复核

3. 发布与SLA
- 发布“对外能力清单 + SLA”
- 值班响应流程、故障分级、演练计划固化
- 交付验收材料归档（报告、日志、看板、脚本）

**退出标准（DoD）**：

- 连续 14 天无 P0
- 性能与安全指标达标
- 对外能力清单与 SLA 已发布并可核验

## 5. 模块级开发工作包（WBS）

## 5.1 智能体与工作流

1. 工作流节点语义补齐（循环/协作/复杂节点）。
2. 执行引擎状态机统一（待执行、运行中、成功、失败、重试中、终止）。
3. `api/workflows/*` 与 `api/v1/workflow/run` 契约统一。
4. 工作流执行 Trace 自动注入与回放。

## 5.2 知识库与检索

1. 文档处理管线标准化：上传、解析、切片、向量化、索引。
2. OCR/ASR 依赖检测、降级策略、失败语义标准化。
3. 图谱链路拆分：构建任务、关系抽取、检索回路、可视化展示。
4. 数据源接入分级：内建稳定源、实验源、预留源。

## 5.3 多智能体协作

1. 任务包协议：输入、子任务、执行策略、汇总结构。
2. 串行/并行调度器与超时控制。
3. 失败恢复：重试、跳过、人工接管。
4. 协作结果可追溯：任务级指标、日志关联、输出版本化。

## 5.4 观测与运维

1. 统一指标字典与埋点规范。
2. Trace、日志、指标关联查询面板。
3. 告警路由与降噪策略。
4. 运维手册、应急预案、故障演练脚本。

## 5.5 网关、安全与系统管理

1. 网关能力重建：鉴权、限流、审计、路由策略。
2. RBAC 权限矩阵与最小权限策略。
3. 统一错误码与用户可读错误信息。
4. 操作审计全覆盖（关键配置、执行、失败事件）。

## 6. 质量工程与发布计划

## 6.1 测试金字塔（最低要求）

1. 单元测试：核心 service/handler 覆盖。
2. 契约测试：前后端字段、错误码、状态码。
3. 集成测试：外部框架连通性、MCP、同步链路。
4. E2E 测试：知识->工作流->协作->观测端到端。
5. Smoke 测试：每次发布前必跑。

## 6.2 发布节奏

- 双周发布（2 周一个迭代）
- 每周固定回归窗口
- 发布门禁：
  - 核心 smoke 全通过
  - P1 缺陷清零
  - 回滚脚本与版本标签齐备

## 6.3 缺陷分级与处理 SLA

- `P0`：4 小时内恢复服务
- `P1`：24 小时内给出修复并上线
- `P2`：下一迭代关闭

## 7. 组织与资源投入（建议）

## 7.1 团队配置

- 后端工程师：3-4（Java/网关/服务）
- 前端工程师：2-3（控制台/观测/协作页面）
- AI 工程师：2（Python 执行器/模型接入）
- 测试工程师：2（自动化/E2E/性能）
- DevOps/SRE：1-2（监控、发布、告警、演练）
- PM/架构：1-2（需求、排期、验收）

## 7.2 关键角色职责

1. 技术负责人：里程碑达成与架构决策。
2. 模块 Owner：对模块 KPI 与验收证据负责。
3. QA Owner：质量门禁与回归稳定性负责。
4. SRE Owner：观测、告警、SLA 与演练负责。

## 8. 风险清单与缓释策略

| 风险 | 影响 | 概率 | 缓释策略 |
| --- | --- | --- | --- |
| 外部依赖波动（Dify/RAGFlow 等） | 集成不稳定、验收延期 | 高 | 建立适配层 + 降级策略 + 健康检查 |
| 页面与后端字段不一致 | 前端报错、统计失真 | 中高 | 契约测试前置，接口版本管理 |
| 协作链复杂导致失败率高 | 关键模块不达标 | 中高 | 先串行保底，再并行扩展，增加回退机制 |
| 观测体系碎片化 | 问题定位慢 | 中 | traceId/taskId 强制贯穿 |
| 测试覆盖不足 | 回归反复 | 中高 | 强制 smoke + E2E 门禁 |
| 人力并行冲突 | 返工与延期 | 中 | 按模块明确 Owner 与分支策略 |

## M3（第 11 周末）

3. Phase 3 完成后：
“ORIN 已达到成熟平台交付标准，支持稳定运维、审计追踪与 SLA 对外承诺。”

## 11. 本周可立即启动的执行清单（第一优先级）

1. 建立“页面-接口-执行器”映射总表并标注责任人。

3. 修复 `runtime/overview`、`runtime/traces` 出数链路并补 trace 查询闭环。

5. 建立三端 smoke 流程并接入 CI 发布门禁。
