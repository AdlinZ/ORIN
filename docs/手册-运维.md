# ORIN 运维手册 · 备份、恢复与灾难应对

> 目标：一线运维/后端工程师能照本文档在 1 小时内从零恢复出可用环境。

## 1. 备份策略

### 1.1 备份内容（`scripts/backup.sh` 覆盖）

| 内容 | 命令 | 落地 |
|------|------|------|
| MySQL 全备 | `mysqldump --single-transaction --routines --events --triggers` | `mysql/orindb.sql.gz` |
| Redis RDB | `redis-cli BGSAVE` + `docker cp` | `redis/dump.rdb` |
| RabbitMQ 定义 | `rabbitmqctl list_exchanges/queues/bindings` | `rabbitmq/{exchanges,queues,bindings}.txt` |
| 配置文件 | `.env` / `docker-compose.override.yml` / `docker-compose.yml` | `config/` |
| 已构建产物 | `orin-backend-1.0.0.jar` / `orin-frontend/dist` | `artifacts/`（可选） |
| 元信息 | 时间戳 / commit / 容器状态 | `metadata.txt` |

### 1.2 保留策略

- 默认保留最近 7 份（`--keep N` 可调）
- 旧备份按时间戳目录名自然淘汰
- 备份目录：`./backups/orin-YYYYMMDD-HHMMSS/`

### 1.3 调度（crontab 推荐）

```cron
# 每日凌晨 3 点全量备份，保留 14 份（覆盖周末 + 缓冲）
0 3 * * * cd /opt/orin && /opt/orin/scripts/backup.sh --keep 14 >> /var/log/orin/backup.log 2>&1
```

### 1.4 异地同步建议

```bash
# 备份后同步到异地（S3 / OSS / 远程盘）
# 推荐 rclone + cron，避免本地磁盘满后无法恢复
0 4 * * * rclone sync /opt/orin/backups/ remote:orin-backups/ --transfers 4
```

## 2. 手动备份

```bash
# 立即全量备份（默认 7 份）
./scripts/backup.sh

# 保留 14 份
./scripts/backup.sh --keep 14

# 验证备份完整性
./scripts/backup-smoke.sh ./backups/orin-20260615-030000
```

## 3. 灾难恢复 SOP

### 3.1 场景 A：单服务挂了

| 现象 | 步骤 |
|------|------|
| MySQL 挂了 | `docker compose up -d mysql`（数据卷 `orin-mysql-data` 持久化） |
| Redis 挂了 | `docker compose up -d redis`（`appendonly yes` 重启即恢复） |
| RabbitMQ 挂了 | `docker compose up -d rabbitmq`（持久化卷存活） |
| AI Engine 挂了 | `docker compose up -d orin-ai-engine` |
| Backend 挂了 | `docker compose up -d orin-backend` |
| Jaeger 挂了 | `docker compose up -d jaeger`（无状态，丢数据不影响业务） |

### 3.2 场景 B：磁盘满

```bash
# 1. 找大文件
du -sh /opt/orin/backups/* | sort -hr | head -5

# 2. 删旧备份（保留期外）
./scripts/backup.sh --keep 3   # 重设保留期，会触发旧备份清理
# 或手动：rm -rf /opt/orin/backups/orin-202605xx-*

# 3. 看 Docker 卷
docker system df

# 4. 清理未用镜像 / 停止的容器
docker image prune -a
docker container prune
```

### 3.3 场景 C：完整灾难恢复（从零开始）

**前置**：新服务器 + 异地备份目录

```bash
# 1. 装 Docker + Docker Compose
curl -fsSL https://get.docker.com | sh
# 安装 compose（按系统，参考 https://docs.docker.com/compose/install/）

# 2. 拉代码
git clone https://github.com/yourname/ORIN.git /opt/orin
cd /opt/orin

# 3. 准备 .env（从备份恢复 .env.backup）
cp /path/to/backup/config/.env.backup .env
# 编辑 .env 改成本地环境（端口、域名、JWT_SECRET）

# 4. 启动基础设施
docker compose up -d mysql redis rabbitmq jaeger

# 5. 等 MySQL 健康
docker compose ps   # 看到 mysql service_healthy 再继续

# 6. 恢复 MySQL
./scripts/restore.sh /path/to/backup --mysql

# 7. 恢复 Redis
./scripts/restore.sh /path/to/backup --redis

# 8. 恢复 RabbitMQ（重新声明 exchanges/queues/bindings 文本）
./scripts/restore.sh /path/to/backup --rabbitmq

# 9. 启动业务服务
docker compose up -d orin-backend orin-ai-engine orin-frontend

# 10. 跑端到端 smoke
./scripts/business-smoke.sh
./scripts/trace-smoke.sh

# 11. 验证
curl -fsS http://localhost:8080/api/v1/health
curl -fsS http://localhost:8000/health
```

### 3.4 场景 D：配置错乱回滚

```bash
# 回滚到指定备份的 config 部分（不影响数据库）
./scripts/rollback.sh /path/to/backup --config-only

# 回滚到上一次完整备份（全量）
./scripts/rollback.sh
```

## 5. 故障分级与应急

| 级别 | 现象 | 响应时间 | 升级 |
|------|------|----------|------|
| P0 | 业务完全不可用 | 15 分钟 | 立即召集 |
| P1 | 核心功能降级（如 AI Engine 慢） | 1 小时 | 当日修复 |
| P2 | 非核心受影响（如 Jaeger 挂） | 4 小时 | 排期内修复 |
| P3 | 监控/告警类 | 24 小时 | 下次迭代 |

## 6. 备份验证

`scripts/backup-smoke.sh` 在每次 backup 后（或 cron 中）跑，验证：
- 备份目录结构完整（mysql/redis/rabbitmq/config/artifacts/metadata.txt）
- MySQL dump 可被 `gunzip` + 头部 `mysqldump` magic 解压
- Redis RDB 是 Redis 持久化格式
- RabbitMQ 三个 .txt 都有内容
- 配置文件可读
- `metadata.txt` 字段完整

**不**实际恢复（避免误覆盖线上数据），**只**做静态完整性检查。

## 8. 升级与回滚

| 操作 | 命令 |
|------|------|
| 升级 | `git pull && docker compose up -d --build` |
| 紧急回退代码 | `git checkout <commit> && docker compose up -d --build` |
| 仅回退 backend | `docker compose up -d --build orin-backend` |
| 仅回退 ai-engine | `docker compose up -d --build orin-ai-engine` |

回退**数据库 schema** 必须先看 `orin-backend/src/main/resources/db/migration/`：
- Flyway 永远向前，不回退
- 若新版 schema 写入了旧版没有的表/列，回退会启动失败 —— 此时应
  - 用 backup.sh 恢复数据库到升级前
  - 再回退代码
- 这是"场景 C"流程的子集

## 10. 启动期密钥校验（Phase 3 起步 · 小刀 8）

`com.adlin.orin.config.SecurityPropertiesValidator` 在 Spring Boot 启动
期对必需 / 占位符密钥做硬校验，**生产 / staging 环境遇占位符直接拒启动**。

### 10.1 占位符黑名单

`.env.example` / 文档 / 早期 commit 残留的占位符值，命中任一 → prod /
staging 抛 `BusinessException` 拒启动：

| 变量 | 占位符黑名单（命中即拒） |
|------|--------------------------|
| `ORIN_DEFAULT_ADMIN_PASSWORD` | `admin123` |
| `DB_PASSWORD` | `replace_with_local_db_password` 等 |
| `REDIS_PASSWORD` | `replace_with_local_redis_password` 等 |
| `RABBITMQ_PASSWORD` | `replace_with_local_rabbitmq_password` 等 |
| `JWT_SECRET` | `your_256_bit_random_secret_key_here`、`your_64_bytes_random_secret_generated_by_openssl_rand_base64_64` |
| 通用前缀 | `replace_with_*` / `changeme*` / `your_*_here` / `todo_*` |

### 10.2 JWT_SECRET 长度

- 长度 < 32 字符（256 bit）→ prod / staging 抛异常拒启动
- dev / test 仅 warn，不阻断

### 10.3 生成强密钥

```bash
# 256 bit JWT_SECRET
openssl rand -base64 32

# 64 bytes 随机
openssl rand -base64 64
```

### 10.4 自检

```bash
# 跑后端测试套件，验证拒绝路径
cd orin-backend
mvn -B -Dtest=SecurityPropertiesValidatorTest test
# 期望：Tests run: 19, Failures: 0, Errors: 0
```

### 10.5 失败排查

启动期抛 `BusinessException` 含 "占位符密钥" / "JWT_SECRET 长度"：
1. 检查 `.env` / `docker-compose.yml` 里对应变量
2. 替换为 `openssl rand -base64 32` 生成的强密钥
3. 重启服务
