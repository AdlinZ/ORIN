# ORIN 三层观测模型

本文档定义 ORIN 系统的三层观测体系架构，明确每层观测的职责、数据类型、核心指标和使用场景。

---

## 1. 三层观测架构概览

```
┌─────────────────────────────────────────────────────────────────┐
│                     ORIN 三层观测体系                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────────────┐   │
│  │  第一层      │   │  第二层      │   │  第三层              │   │
│  │  Prometheus │   │  Langfuse   │   │  ORIN Audit/Trace   │   │
│  │  基础设施监控 │   │  LLM追踪    │   │  业务审计日志         │   │
│  └──────┬──────┘   └──────┬──────┘   └──────────┬──────────┘   │
│         │                  │                      │              │
│         ▼                  ▼                      ▼              │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────────────┐   │
│  │ CPU/内存    │   │ Trace/Span  │   │ API调用记录           │   │
│  │ 磁盘/网络   │   │ Generation  │   │ 对话日志              │   │
│  │ GPU指标     │   │ Tool执行    │   │ 工作流执行            │   │
│  └─────────────┘   └─────────────┘   └─────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 第一层：Prometheus（基础设施监控）

### 2.1 职责定位
**核心职责**：监控主机和容器层面的基础设施指标，包括 CPU、内存、磁盘、网络、GPU 等硬件资源使用情况。

### 2.2 数据类型
| 指标类型 | 具体指标 | 数据格式 |
|---------|---------|---------|
| CPU | 使用率、负载、核心数、型号 | 百分比 (0-100) |
| 内存 | 使用率、总容量、可用容量 | 字节数 |
| 磁盘 | 使用率、总容量、分区信息 | 百分比 (0-100) |
| 网络 | 流入/流出速率 | Bytes/s |
| GPU | 使用率、显存使用率、显存总量、GPU型号 | 百分比或字节数 |

### 2.3 核心服务
- **服务类**：`PrometheusService`
- **任务类**：`ServerHardwareMonitorTask`、`SystemHealthCheckTask`
- **实体类**：`ServerHardwareMetric`、`ServerInfo`

### 2.4 使用场景
- 系统容量规划
- 性能瓶颈定位
- 告警阈值设置（如 CPU > 80% 触发告警）
- 运维大盘展示

### 2.5 配置项
```yaml
# Prometheus 连接配置
prometheus:
  endpoint: http://localhost:9090  # Prometheus 服务器地址

# 系统监控任务配置
monitor:
  hardware:
    enabled: true
    interval: 30000  # 采集间隔(ms)
```

## 3. 第二层：Langfuse（LLM 追踪）

### 3.1 职责定位
**核心职责**：追踪 AI/LLM 调用链路，记录提示词、响应、Token 消耗、延迟等 AI 特有的指标，用于 AI 行为分析和成本控制。

### 3.2 数据类型
| 追踪类型 | 说明 | 典型字段 |
|---------|------|---------|
| Trace | 一次完整的 AI 请求追踪 | id, name, metadata, timestamps |
| Generation | LLM 生成记录 | model, prompt, completion, tokens, latency |
| Span | 工具调用或中间步骤 | name, startTime, endTime, input, output |
| Event | 关键业务事件 | name, metadata |
| Score | 评估分数 | name, value, comment |

### 3.3 核心服务
- **服务类**：`LangfuseObservabilityService`
- **切面类**：`LangfuseTracingAspect`
- **包装类**：`LangfuseTracingWrapper`

### 3.4 使用场景
- AI 调用链路可视化（类似 OpenTelemetry 追踪）
- Token 消耗统计与成本分摊
- Prompt 工程效果评估
- AI 响应质量评分
- 慢请求分析与优化

### 3.5 配置项
```yaml
# Langfuse 配置
langfuse:
  enabled: false                    # 是否启用
  public-key: ${LANGFUSE_PUBLIC_KEY} # 公钥
  secret-key: ${LANGFUSE_SECRET_KEY} # 密钥
  host: ${LANGFUSE_HOST:}            # 自托管地址，默认使用云服务
```

## 4. 第三层：ORIN Audit/Trace（业务审计日志）

### 4.1 职责定位
**核心职责**：记录业务层面的操作审计日志，包括 API 调用、用户操作、智能体对话、工作流执行等业务事件，用于合规审计和业务分析。

### 4.2 数据类型
| 审计类型 | 说明 | 典型字段 |
|---------|------|---------|
| API 调用 | HTTP API 请求响应 | userId, endpoint, method, statusCode, responseTime |
| 对话日志 | 智能体对话记录 | conversationId, agentId, query, response, tokens |
| 工作流追踪 | 工作流执行步骤 | traceId, instanceId, stepId, status, duration |
| 协作事件 | 协作任务状态变更 | packageId, subtaskId, eventType, result |

### 4.3 核心服务
- **审计服务**：`AuditLogService`、`AuditHelper`
- **追踪服务**：`TraceService`
- **实体类**：`AuditLog`、`WorkflowTraceEntity`、`ConversationLog`

### 4.4 使用场景
- 合规审计（谁在什么时间做了什么操作）
- 业务数据分析（用户行为、功能使用频率）
- 问题排查与故障定位
- 多系统关联分析（通过 traceId 串联）

### 4.5 核心字段说明
```java
// AuditLog 核心字段
AuditLog {
    userId          // 用户ID
    providerId      // 智能体/Provider ID
    providerType     // Provider 类型 (DIFY, SiliconFlow, 等)
    traceId         // 追踪ID，串联整个调用链
    conversationId  // 对话ID
    endpoint        // API 端点
    method          // HTTP 方法
    requestParams   // 请求参数
    responseContent // 响应内容
    statusCode      // HTTP 状态码
    responseTime    // 响应时间(ms)
    promptTokens    // 输入 Token 数
    completionTokens// 输出 Token 数
    totalTokens     // 总 Token 数
    success         // 是否成功
    errorMessage    // 错误信息
    workflowId      // 工作流ID
    fileId          // 文件ID(多模态)
    downloadUrl     // 下载URL
}
```

## 5. 三层联动机制

### 5.1 traceId 串联
- 三层观测通过统一的 `traceId` 进行关联
- `traceId` 通过 MDC（Mapped Diagnostic Context）传递
- 请求入口处生成，贯穿整个请求生命周期

```
请求入口 → traceId 生成 → Prometheus（基础指标附加traceId）
                      → Langfuse（LLM追踪附加traceId）
                      → AuditLog（业务日志附加traceId）
```

### 5.2 告警源定义

ORIN 系统告警分为以下四大类，通过 `AlertService.triggerSystemAlert()` 统一触发：

#### 5.2.1 系统资源告警 (SYSTEM_RESOURCE)

**触发条件**：系统资源使用率超过阈值

| 资源类型 | 告警规则类型 | 默认阈值 | 触发方式 |
|---------|-------------|---------|---------|
| CPU | `SYSTEM_RESOURCE` | > 80% | Prometheus 采集触发 |
| 内存 | `SYSTEM_RESOURCE` | > 85% | Prometheus 采集触发 |
| 磁盘 | `SYSTEM_RESOURCE` | > 90% | Prometheus 采集触发 |
| GPU | `SYSTEM_RESOURCE` | > 90% | Prometheus 采集触发 |

**告警调用示例**：
```java
alertService.triggerSystemAlert("SYSTEM_RESOURCE", "CPU", "CPU 使用率超过 80%: 当前 " + cpuUsage + "%");
```

#### 5.2.2 调用失败告警 (ERROR_RATE)

**触发条件**：外部 AI API 调用失败

| 失败类型 | 告警规则类型 | 触发方式 |
|---------|-------------|---------|
| API 调用失败 (HTTP 5xx) | `ERROR_RATE` | AuditLogService 自动触发 |
| API 超时 | `ERROR_RATE` | AuditLogService 自动触发 |
| Provider 连接失败 | `ERROR_RATE` | AuditLogService 自动触发 |

**告警调用示例**：
```java
// AuditLogService 中已实现
alertService.triggerSystemAlert("ERROR_RATE", providerId, "API调用失败: " + errorMsg);
```

#### 5.2.3 协作任务告警 (COLLABORATION_FAILURE)

**触发条件**：协作任务执行异常

| 失败类型 | 告警规则类型 | 触发方式 |
|---------|-------------|---------|
| 任务包执行失败 | `COLLABORATION_FAILURE` | CollaborationExecutor |
| 子任务失败 | `COLLABORATION_FAILURE` | CollaborationExecutor |
| 共识达成失败 | `COLLABORATION_FAILURE` | CollaborationOrchestrator |
| 回退触发 | `COLLABORATION_FAILURE` | CollaborationOrchestrator |

**告警调用示例**：
```java
alertService.triggerSystemAlert("COLLABORATION_FAILURE", packageId, "协作任务包执行失败: " + errorMsg);
```

#### 5.2.4 同步失败告警 (SYNC_FAILURE)

**触发条件**：外部数据同步异常

| 失败类型 | 告警规则类型 | 触发方式 |
|---------|-------------|---------|
| 知识库同步失败 | `SYNC_FAILURE` | KnowledgeTaskListener |
| Dify/RAGFlow 同步失败 | `SYNC_FAILURE` | SideClientSyncController |
| 外部 API 同步超时 | `SYNC_FAILURE` | SideClientSyncController |
| 向量化失败 | `SYNC_FAILURE` | KnowledgeTaskListener |

**告警调用示例**：
```java
alertService.triggerSystemAlert("SYNC_FAILURE", taskId, "知识库同步失败: " + errorMsg);
```

#### 5.2.5 系统健康告警 (SYSTEM_HEALTH)

**触发条件**：依赖的外部服务不可用

| 服务类型 | 告警规则类型 | 触发方式 |
|---------|-------------|---------|
| MySQL 连接失败 | `SYSTEM_HEALTH` | SystemHealthCheckTask |
| Redis 连接失败 | `SYSTEM_HEALTH` | SystemHealthCheckTask |
| Milvus 连接失败 | `SYSTEM_HEALTH` | SystemHealthCheckTask |
| 外部 API Key 无效 | `SYSTEM_HEALTH` | HealthCheckTask |

#### 5.2.6 告警规则类型汇总

| 规则类型 | 说明 | 严重程度 |
|---------|------|---------|
| `SYSTEM_RESOURCE` | 系统资源超阈值 | WARNING |
| `ERROR_RATE` | API 调用失败 | ERROR |
| `COLLABORATION_FAILURE` | 协作任务失败 | ERROR |
| `SYNC_FAILURE` | 同步操作失败 | WARNING |
| `SYSTEM_HEALTH` | 系统健康检查失败 | CRITICAL |

### 5.3 统一通知渠道抽象

告警通知通过 `NotificationService.sendNotification()` 统一发送，支持以下渠道：

| 渠道 | 配置值 | 实现类 |
|------|-------|-------|
| 邮件 | `EMAIL` | `NotificationService.sendEmail()` |
| 钉钉 | `DINGTALK` | `NotificationService.sendDingTalk()` |
| 企业微信 | `WECHAT` | `NotificationService.sendWeChat()` |

**使用方式**：
```java
// 触发告警（自动发送通知）
alertService.triggerSystemAlert("ERROR_RATE", providerId, "API调用失败");

// 或创建规则时指定通知渠道
AlertRule rule = AlertRule.builder()
    .ruleName("API失败告警")
    .ruleType("ERROR_RATE")
    .notificationChannels("EMAIL,DINGTALK")  // 逗号分隔多渠道
    .recipientList("admin@example.com,https://oapi.dingtalk.com/...")
    .build();
```

### 5.4 配置统一入口
三层观测的配置通过 `ObservabilityController` 统一管理：
- `GET /api/v1/observability/langfuse/status` - Langfuse 状态
- `GET /api/v1/observability/config` - 观测配置信息
- `GET /api/v1/observability/dashboard/unified` - 统一大盘入口

## 6. 前端观测面板

### 6.1 前端职责边界

**前端只做统一入口和摘要展示**，具体原则如下：

| 前端应该 | 前端不应该 |
|---------|-----------|
| 显示统计摘要（成功率、延迟分布、Token 消耗） | 渲染详细链路追踪（Span/Event 详情） |
| 提供外部面板链接（Langfuse Dashboard） | 尝试解析 Langfuse 追踪数据 |
| 展示关键指标卡片（CPU/内存/请求量） | 替代 Prometheus 绘制详细时序图 |
| 提供 traceId 查询入口和关联展示 | 直接展示完整的对话内容或文件 |
| 聚合展示审计日志摘要 | 解析和展示完整的请求/响应 JSON |

### 6.2 统一观测入口

前端通过以下 API 获取各层观测的统一入口信息：

```
GET /api/v1/observability/dashboard/unified
```

返回结构：
```json
{
  "prometheus": {
    "endpoint": "/api/v1/monitor/dashboard/summary",
    "description": "系统监控大盘"
  },
  "langfuse": {
    "enabled": true,
    "link": "https://cloud.langfuse.com"
  },
  "trace": {
    "endpoint": "/api/traces",
    "description": "业务追踪记录"
  }
}
```

### 6.3 已实现的前端面板

| 面板文件 | 职责 |
|---------|------|
| `MonitorDashboard.vue` | 统一监控入口，显示统计摘要、traceId 查询 |
| `ServerMonitor.vue` | Prometheus 指标卡片（CPU/内存/磁盘/GPU） |
| `TokenStats.vue` | Token 消耗统计 |
| `LatencyStats.vue` | 延迟分布统计 |
| `CostStats.vue` | 成本统计 |
| `ErrorStats.vue` | 错误率统计 |
| `TaskQueue.vue` | 异步任务状态 |

## 8. 必填字段规范

本节明确后端必须写入各层观测数据的字段规范。

### 8.1 Prometheus 必填指标

所有主机监控指标均为**自动采集**，无需手动指定。

| 指标分类 | 必填字段 | 说明 |
|---------|---------|------|
| 系统信息 | hostname, os, cpuModel, cpuCores | 基础信息 |
| CPU | cpuUsage | CPU 使用率百分比 |
| 内存 | memoryUsage, memoryTotal | 使用率和总量(字节) |
| 磁盘 | diskUsage | 磁盘使用率百分比 |
| 网络 | networkReceiveRate, networkTransmitRate | 速率(字节/秒) |
| GPU | gpuUsage, gpuMemoryUsage, gpuMemoryTotal | GPU 相关指标 |

### 8.2 Langfuse 必填字段

LLM 追踪通过切面自动采集，调用 `LangfuseObservabilityService` 方法时必须传入：

| 追踪类型 | 必填字段 | 可选字段 | 说明 |
|---------|---------|---------|------|
| Generation | traceId, model, promptTokens, completionTokens, totalTokens, latencyMs | completion | LLM 生成记录 |
| Event | traceId, name, metadata | - | 业务事件 |
| Span | traceId, name, startTime, endTime | input, output, metadata | 工具调用 |

**注意**：
- `traceId` 必须有效且非空
- `prompt` 字段因数据量大，默认不记录完整内容
- Langfuse 未启用时，所有调用自动降级，不影响主流程

### 8.3 ORIN Audit 必填字段

业务审计通过 `AuditLogService.logApiCall()` 或 `AuditHelper` 方法写入，必须包含：

| 字段分类 | 必填字段 | 可选字段 | 说明 |
|---------|---------|---------|------|
| 身份识别 | userId, providerId, providerType | - | 标识调用者和调用目标 |
| 追踪关联 | traceId | conversationId, workflowId | 用于关联各层数据 |
| 请求信息 | endpoint, method, requestParams | - | 请求详情 |
| 响应信息 | statusCode, responseTime, success | responseContent, errorMessage | 响应详情 |
| Token统计 | promptTokens, completionTokens, totalTokens | estimatedCost | AI 特有指标 |
| 多媒体 | - | fileId, downloadUrl | 多模态文件信息 |

### 8.4 各服务必填字段对照表

| 服务/方法 | Audit 字段 | Langfuse 字段 |
|----------|-----------|--------------|
| AgentManageServiceImpl.chat() | userId, agentId, traceId, conversationId, endpoint, model, tokens, responseTime | traceId, model, tokens, latencyMs |
| batchExportAgents() | userId, agentId, operation, exportCount, success | - |
| batchImportAgents() | userId, agentId, operation, importCount, skippedCount, success | - |
| refreshAllAgentsMetadata() | userId, agentId, operation, successCount, failedCount, success | - |
| WorkflowProxyController.run() | traceId, workflowId, instanceId, endpoint | traceId, model, tokens, latencyMs |
| CollaborationOrchestrator | traceId, packageId, subtaskId, eventType | traceId, event name |

### 8.5 字段降级规则

当某层观测不可用时的降级处理：

| 观测层不可用 | 降级行为 | 是否影响主流程 |
|------------|---------|--------------|
| Prometheus 不可达 | 使用本地 MXBean 指标 | 否 |
| Langfuse 未启用 | 跳过追踪记录 | 否 |
| Audit 日志写入失败 | 记录本地日志，降级不抛异常 | 否 |
