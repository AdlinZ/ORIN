智能体模块已完成**接入、对话、多Provider路由、批量导入导出、元数据刷新、异步任务**的端到端实现。以下按能力逐项标注验收状态。

## 一、已验收能力（✅）

### 1.1 智能体接入（onboardAgent）

| Provider | 连接测试 | 元数据获取 | API Key 处理 | 状态 |
|----------|---------|-----------|-------------|------|
| Dify | ✅ testConnection | ✅ fetchAppMeta + fetchAppParameters | 明文存储 | 可交付 |
| SiliconFlow | ✅ testConnection | ⚠️ 默认值（无远程获取） | 明文存储 | 可交付 |
| Ollama | ✅ testConnection（含模型验证） | ⚠️ 本地模型 | 明文存储 | 可交付 |
| MiniMax | ✅ testConnection（含模型验证） | ⚠️ 默认值 | 明文存储 | 可交付 |
| Kimi/Moonshot | ❌ 未实测 | ❌ 无 | ❌ 未接入 | 不可用 |
| Zhipu/GLM | ❌ 未实测 | ❌ 无 | ❌ 未接入 | 不可用 |
| DeepSeek | ❌ 未实测 | ❌ 无 | ❌ 未接入 | 不可用 |

**Provider 自动识别**：URL 包含 `siliconflow` → SiliconFlow；`ollama` / `11434` → Ollama；`deepseek` → DeepSeek；`minimax` → MiniMax；其他默认 Dify。

**接入流程**：
1. identifyProvider() 识别 Provider
2. testConnection() 验证连通性（**失败则抛异常，不允许接入**）
3. 保存 AgentAccessProfile（endpoint + apiKey + 连接状态）
4. 保存 AgentMetadata（name + modelName + providerType + viewType）
5. 初始化 AgentHealthStatus

**已知风险**：
- API Key 明文存储在数据库，无加密
- SiliconFlow 接入时不校验 Key，只测连通性

### 1.2 聊天对话（chat）

**多 Provider 路由**（chatWithConversation 方法）：

| viewType | SiliconFlow | Ollama | Dify | Kimi | Zhipu | DeepSeek | MiniMax |
|----------|------------|--------|------|------|-------|---------|---------|
| CHAT（默认） | ✅ chat() | ✅ chat() | ✅ sendMessage() | ✅ chat() | ✅ chat() | ✅ chat() | ✅ chat() |
| TTI / TEXT_TO_IMAGE | ✅ generateImage() | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| TTS / TEXT_TO_SPEECH | ✅ generateAudio() | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| STT / SPEECH_TO_TEXT | ✅ transcribeAudio() | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| TTV / TEXT_TO_VIDEO | ✅ generateVideo()（30s轮询，最长15分钟） | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| 其他 | ✅ 降级 chat() | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |

**对话链路**：
1. 读取 AgentAccessProfile + AgentMetadata
2. 组装 SystemPrompt（MetaKnowledgeService → 长期记忆 → Skills → PromptTemplate）
3. 追加最近 10 条会话历史（AuditLog）
4. 调用 Provider API
5. 提取响应文本（支持 OpenAI 格式 choices[0].message.content + reasoning_content）
6. 保存 ConversationLog + AuditLog（原始响应 JSON）
7. 返回 enrichment 后的响应（conversation_id + viewType + provider + model）

**图片生成后处理**：SiliconFlow 返回的图片 URL 会下载到本地 MultimodalFile，替换为 `/api/v1/multimodal/files/{id}/download`。

**视频生成**：提交任务后每 30s 轮询，最多重试 180 次（约 15 分钟）。

### 1.3 批量导出（batchExportAgents）

- ✅ 导出所有智能体（传入 null）或指定 ID 列表
- ✅ API Key 全程 `***MASKED***`，不泄露
- ✅ 导出 JSON 包含：metadata + accessProfile（不含明文 Key）
- ✅ 有 AuditLog 记录

**风险**：导出文件在内存中拼接，大规模（>1000智能体）可能有 OOM 风险。

---

### 1.4 批量导入（batchImportAgents）

- ✅ 跳过 agentId 为 null 的无效条目
- ✅ 跳过已存在的 agentId（不覆盖）
- ✅ 无 accessProfile 时只创建 metadata（Key 需重新配置）
- ✅ 批量导入有审计日志
- ⚠️ **导入时 API Key 为空**：因为导出时 Key 被 mask，导入后需要用户重新配置

## 二、伪完成能力（⚠️）

以下能力有代码入口，但实际不可用或行为不符合预期：

### 2.1 MiniMax/Kimi/Zhipu/DeepSeek Chat

代码中有 `kimiAgentManageService.chat()`、`zhipuAgentManageService.chat()` 等调用链路，但这些 service 是否真实存在且已实现需要验证。

**验证方式**：启动应用，创建对应 Provider 的智能体，发送消息，观察是否返回真实响应。

### 2.2 API Key 安全

所有 Provider 的 API Key 以明文存储在 `AgentAccessProfile.api_key` 字段中，无加密。

**风险**：数据库泄漏 = 所有 Provider Key 泄漏。

### 2.3 权限控制

当前没有用户-智能体的权限映射：任何登录用户可以访问/编辑/删除任意智能体。
