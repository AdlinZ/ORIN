# 协作执行架构约束（唯一执行链）

## 目标
消除 "LangGraph / MQ Worker / executor" 并存导致的执行语义分裂，确保协作任务只有一条执行链。

## 唯一执行链
1. 编排层：`app.engine.collaboration_langgraph`
2. 分发/消费层：`app.engine.mq_worker`
3. 执行内核：`app.engine.task_runtime.TaskRuntime`

## 职责边界
- `collaboration_langgraph`
  - 负责状态机、路由、重试决策、共识流程。
  - 不实现具体任务执行细节。
- `mq_worker`
  - 负责 MQ 消费、幂等、结果回写。
  - 不实现任务执行逻辑。
- `task_runtime`
  - 负责单任务执行（Agent/Workflow）。
  - 是协作执行的唯一计算内核。
- `executor.py`
  - 保留为通用工作流执行能力（非协作编排）。
  - 不作为协作流程的编排入口。

## 明确禁止
- 在 `mq_worker` 中新增或恢复独立执行逻辑。
- 在 LangGraph 节点中新增第二套执行引擎。
- 新增并维护与 `task_runtime` 并行的第三套执行内核。

## 兼容策略
- `collaboration_executor.py` 仅保留兼容导入用途。
- `execute_collaboration` 已废弃，统一使用 `run_collaboration`。

## 变更检查清单（PR 必检）
- 协作执行是否仍经由 `TaskRuntime`。
- 是否引入新的任务执行实现而未复用 `TaskRuntime`。
- 是否新增了第二套 worker 队列协议或独立消费环。
- 协作 API 的 worker 启停是否仍指向统一 `mq_worker`。
