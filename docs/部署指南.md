# ORIN 部署指南

涵盖本地开发与生产部署，整合环境变量、依赖检查、常见坑位。

## 1. 环境要求

| 组件 | 版本 | 必需 |
|------|------|------|
| JDK | 17+ | 是 |
| Node.js | 18+ | 是 |
| Python | 3.11+ | 是（AI 引擎） |
| MySQL | 8.0+ | 是 |
| Redis | 6.0+ | 是（最小闭环） |
| Milvus | 2.3+ | 仅向量检索 |
| RabbitMQ | 3.x | 仅协作队列 |
| FFmpeg / Tesseract | — | 仅多模态降级 |

端口分配：

| 服务 | 端口 |
|------|------|
| orin-backend | 8080 |
| orin-frontend | 5173 |
| orin-ai-engine | 8000 |
| MySQL | 3306 |
| Redis | 6379 |
| Milvus | 19530 |

## 2. 数据库初始化

```sql
CREATE DATABASE orindb CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'orin_user'@'localhost' IDENTIFIED BY 'your_secure_password';
GRANT ALL PRIVILEGES ON orindb.* TO 'orin_user'@'localhost';
FLUSH PRIVILEGES;
```

JPA 在 dev 环境为 `update` 模式，首次启动自动建表；生产环境应改为 `validate` 并使用迁移脚本。

## 3. 环境变量

### 3.1 必需变量（生产）

| 变量 | 说明 |
|------|------|
| `SPRING_PROFILES_ACTIVE` | 设为 `prod` |
| `DB_HOST` / `DB_PORT` / `DB_NAME` | 数据库连接 |
| `DB_USERNAME` / `DB_PASSWORD` | 数据库凭据（生产用专用账号） |
| `JWT_SECRET` | JWT 签名密钥（≥256 位，`openssl rand -base64 64` 生成） |
| `REDIS_HOST` / `REDIS_PORT` / `REDIS_PASSWORD` | Redis 连接 |
| `CORS_ALLOWED_ORIGINS` | 生产域名白名单，禁止 `*` |

### 3.2 可选变量

| 变量 | 默认 | 用途 |
|------|------|------|
| `SERVER_PORT` | 8080 | 后端端口 |
| `JWT_EXPIRATION` | 86400000 | Token 过期（毫秒） |
| `MILVUS_HOST` / `MILVUS_PORT` | localhost / 19530 | 向量库 |
| `SILICONFLOW_API_KEY` | — | 默认 LLM provider |
| `DIFY_DEFAULT_ENDPOINT` | — | Dify 集成 |
| `ORIN_MQ_WORKER_AUTO_START` | false | 协作队列自启动 |
| `COLLAB_LISTENERS_AUTO_STARTUP` | true | 后端协作结果队列监听器自启动；本地跑 MQ smoke 时需保持 true |
| `RATE_LIMIT_REQUESTS` / `RATE_LIMIT_PERIOD` | 100 / 60 | 限流 |
| `SPRING_MAIL_*` | — | 邮件通知 |

### 3.3 配置加载顺序

- **dev**：`orin-backend/.env` 自动加载（通过 `EnvFileLoader`），适合本地
- **test**：`application-test.properties` 内嵌配置 + H2 内存库
- **prod**：必须通过系统环境变量或 Docker/K8s Secret 注入

## 4. 本地开发

### 4.1 本机进程模式最小闭环

当前优先支持的本机开发基线是“三端本机进程 + Redis + MySQL 或等价可达数据库连接”：

- 必需进程：`orin-backend`、`orin-frontend`、`orin-ai-engine`
- 必需基础依赖：Redis、MySQL 或后端可连接的数据库
- 可选增强依赖：Milvus、RabbitMQ、Langfuse、Prometheus/Grafana、Neo4j、MinIO
- 外部服务：模型 provider、Dify/RAGFlow、远程 MCP Server、生产数据库

最小闭环只要求三端 health、前端首页、Redis 与后端数据库连接可用。后端数据库连通性以后端 `GET /api/v1/health` 返回的 `database.status` 为准；`lsof :3306` 只说明本机 TCP 端口是否监听，不代表后端数据库一定不可达。AI Engine 如果返回 `degraded` 且只涉及 RabbitMQ、Milvus 等可选增强依赖，不阻塞本机最小闭环。

手工启动顺序建议：

```bash
# 1. 确认 Redis 与数据库连接
redis-cli ping

# 2. 后端
cd orin-backend && mvn spring-boot:run

# 3. AI Engine
cd orin-ai-engine && venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000

# 4. 前端
cd orin-frontend && npm run dev
```

验证入口：

```bash
curl -fsS http://localhost:8080/v1/health
curl -fsS http://localhost:8080/api/v1/health
curl -fsS http://localhost:8000/health
curl -fsS http://localhost:8000/v1/health
curl -fsS http://localhost:5173/
bash scripts/smoke-test.sh
```

`scripts/smoke-test.sh` 是“本机最小闭环 + 可选 live smoke”，不是完整 E2E。服务未启动时 live smoke 会提示跳过和启动命令；编译、前端 build 或 targeted 单元测试失败时仍会失败退出。

### 4.2 Docker Compose 最小闭环

根目录 `docker-compose.yml` 面向“新 clone 后快速跑通”场景，只启动最小闭环：

- `mysql`
- `redis`
- `orin-backend`
- `orin-ai-engine`
- `orin-frontend`

```bash
cd /path/to/ORIN
cp .env.example .env
docker compose up
```

如果当前机器没有可用 Docker runtime，可以先做静态预检：

```bash
bash scripts/check-docker-quickstart.sh
```

该脚本只检查根 compose 的服务名、MySQL 初始化挂载、schema snapshot 边界和关键服务连线，不启动容器，也不代表 Docker quickstart 已真实可用。有 Docker runtime 后仍必须执行 `docker compose up --build` 并跑 health / smoke 验证。

对外端口：

- 前端：<http://localhost:5173>
- 后端：<http://localhost:8080>
- AI 引擎：<http://localhost:8000>

开发热重载可叠加：

```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up
```

`docker/local-infra/docker-compose.yml` 保持为增强基础设施栈，按需启动 Milvus、Prometheus、Grafana、RabbitMQ、Langfuse、Neo4j、MinIO 等服务。需要验证向量检索、MQ 协作、图谱或可观测链路时，再进入 `docker/local-infra` 并启用对应 profile；不要把它当作根 compose 的必需部分。

### 4.3 知识图谱本地 smoke

知识图谱构建依赖文档解析结果和 LLM 抽取能力。本地手工验证时，先确认后端、MySQL、文件存储和 `SILICONFLOW_API_KEY` 可用，再按下面顺序跑：

```json
{
  "name": "kg-smoke",
  "description": "local knowledge graph smoke",
  "type": "UNSTRUCTURED"
}
```

1. `POST /api/v1/knowledge` 创建知识库。`KnowledgeBase.type` 使用 `UNSTRUCTURED`、`RAGFLOW`、`STRUCTURED` 等枚举值，不使用旧的 `LOCAL`；请求体不要传 `enabled`。
2. `POST /api/v1/knowledge/{kbId}/documents/upload` 上传短文本或 Markdown 文档。
3. 轮询 `kb_documents`，等目标文档进入 `parse_status=PARSED` 且 `vector_status=SUCCESS` 后再触发图谱构建。过早 build 会因正文未准备好而跳过文档。
4. `POST /api/v1/knowledge/graphs/{graphId}/build` 触发构建，轮询 `GET /api/v1/knowledge/graphs/{graphId}` 到 `SUCCESS` 或 `FAILED`。
5. 通过 `knowledge_graphs.entity_count/relation_count` 与 `graph_entities`、`graph_relations` 表行数交叉确认。规范类文档可能抽出实体但没有关系，`relation_count=0` 不一定代表链路失败。

### 4.4 Docker 快照启动机制

根目录 Docker quickstart 使用 `docker/mysql/init/01-orin-schema.sql` 初始化 MySQL。该文件是当前正式支持的新环境 baseline schema snapshot，不是业务数据备份，也不是纯 Flyway 空库重放入口。

当前口径：

- `01-orin-schema.sql` 覆盖 `V1..V83` 的表结构与 `flyway_schema_history`。
- 后端启动后保留 Flyway，由 Flyway 继续执行 `V84..V87`。
- 后续真实 schema 变更从 `V88` 开始新增迁移文件。
- 空库直接重放 `V1..V87` 暂不作为支持路径；部分已发布早期迁移（如 `V5/V6/V8/V11`）依赖历史 JPA 建表结果，不应为了 quickstart 直接改写。

可以用只读脚本确认当前 baseline 与迁移边界：

```bash
bash scripts/check-schema-baseline.sh
```

也可以用 Docker quickstart 静态预检一次性确认 compose 与 snapshot 边界：

```bash
bash scripts/check-docker-quickstart.sh
```

当前快照由开发者本机“已迁移完成”的 MySQL dump 生成。生成脚本为：

```bash
SCHEMA_SNAPSHOT_HOST=localhost \
SCHEMA_SNAPSHOT_USER=root \
SCHEMA_SNAPSHOT_PASS=... \
SCHEMA_SNAPSHOT_DB=orindb \
scripts/regenerate-schema-snapshot.sh
```

脚本只导出 schema 与 `flyway_schema_history` 数据，不导出业务表数据；执行前会要求确认目标数据库，避免误连生产。只有在明确决定重置 baseline snapshot 覆盖范围时才重新生成快照；日常新增 schema 变更应继续走快照之后的 Flyway 迁移。

新增 schema 变更时：

1. 编写新的 `V<N>__xxx.sql` 迁移文件（当前下一次 schema 迁移从 `V88` 起）
2. 至少在一个已迁移到 `V<N-1>` 的环境上跑通 `V<N>`
3. 不改写已发布迁移，尤其不要直接修改 `V5/V6/V8/V11/V87`
4. 仅当需要提升 baseline 覆盖范围时，单独评审并运行 `scripts/regenerate-schema-snapshot.sh` 更新快照

### 4.5 manage.sh 一键启动

```bash
cd /path/to/ORIN
cp orin-backend/.env.example orin-backend/.env   # 编辑数据库与 JWT
./manage.sh start
```

`manage.sh` 行为：

- 检查并启动本地 MySQL（macOS Homebrew）
- 编译后端，启动到 `:8080`
- 启动 AI 引擎到 `:8000`（要求 `orin-ai-engine/venv` 已就绪）
- 启动前端到 `:5173`
- 用真实健康检查路径校验三端，不仅看端口

支持环境覆盖：`ORIN_DB_NAME` / `ORIN_DB_USER` / `ORIN_DB_PASS`。

### 4.6 手动启动

```bash
# 后端
cd orin-backend && mvn spring-boot:run

# AI 引擎（首次需建虚拟环境）
cd orin-ai-engine
python3 -m venv venv
venv/bin/python -m pip install --upgrade pip setuptools wheel
venv/bin/python -m pip install -e .
venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000

# 前端
cd orin-frontend && npm install && npm run dev
```

## 5. 生产部署

### 5.1 构建产物

```bash
cd orin-backend && mvn clean package -DskipTests
cd ../orin-frontend && npm install && npm run build
cd ../orin-ai-engine && python -m venv venv && venv/bin/pip install -e .
```

### 5.2 Systemd 示例

```ini
# /etc/systemd/system/orin-backend.service
[Unit]
Description=ORIN Backend
After=network.target mysql.service

[Service]
Type=simple
User=orin
WorkingDirectory=/opt/orin/orin-backend
EnvironmentFile=/opt/orin/.env
ExecStart=/usr/bin/java -jar /opt/orin/orin-backend/target/orin-backend-1.0.0-SNAPSHOT.jar
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

```ini
# /etc/systemd/system/orin-ai-engine.service
[Unit]
Description=ORIN AI Engine
After=network.target

[Service]
Type=simple
User=orin
WorkingDirectory=/opt/orin/orin-ai-engine
ExecStart=/opt/orin/orin-ai-engine/venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

### 5.3 Nginx 反向代理

```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        root /opt/orin/orin-frontend/dist;
        try_files $uri $uri/ /index.html;
    }

    location ~ ^/(api|v1)/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## 6. 上线前检查清单

### 安全

- [ ] `JWT_SECRET` 为强随机密钥，非默认值
- [ ] `DB_PASSWORD` / `REDIS_PASSWORD` 为强密码（≥16 位）
- [ ] `CORS_ALLOWED_ORIGINS` 配置为实际域名，禁止 `*`
- [ ] `.env` 已加入 `.gitignore`，未提交至代码仓库
- [ ] API Key 以 hash 存储，非明文
- [ ] 生产 JPA 模式为 `validate`

### 配置

- [ ] `SPRING_PROFILES_ACTIVE=prod`
- [ ] 数据库使用独立账号，权限最小化
- [ ] Actuator 端点限制暴露（仅 `health`、`info`）
- [ ] 日志级别为 `WARN` 或 `INFO`

### 联通

- [ ] 三端健康检查通过
- [ ] 至少一个 LLM provider API Key 可用
- [ ] 烟测脚本 `scripts/smoke-test.sh` 通过

## 7. 多模态依赖说明

OCR / ASR 默认走 SiliconFlow VLM；外部依赖未配置时按"降级返回"策略：

| 能力 | 默认 provider | 降级返回 |
|------|---------------|----------|
| 图片 OCR | SiliconFlow VLM | `[OCR Error] SiliconFlow API key not configured` |
| 音频 ASR | SiliconFlow ASR / 本地 Whisper | `[ASR Error] ...` |

启用本地 Whisper：

```properties
whisper.cli.enabled=true
whisper.cli.path=/usr/local/bin/whisper
```

## 8. 常见坑位

1. **`manage.sh` 默认假设 macOS + 本地 MySQL**，Linux 服务器请用手动方式
2. **AI 引擎端口是 8000**，旧文档若提到 8088 已过期
3. **Swagger 路径**：使用 `/swagger-ui/index.html`，旧 `/swagger-ui.html` 已废弃
4. **AI 引擎依赖 `pyproject.toml`**，不再维护 `requirements.txt`
5. **RabbitMQ 本地非强依赖**：未启动时 `mq_worker` 不会自启，健康检查会暴露依赖状态；本地验证协作 MQ smoke 时需同时设置后端 `COLLAB_LISTENERS_AUTO_STARTUP=true` 与 AI 引擎 `ORIN_MQ_WORKER_AUTO_START=true`
6. **多端口并存的健康检查**：脚本用真实路径校验，不应只判断端口占用

## 9. Docker 与 K8s

```yaml
# docker-compose.yml 示例
services:
  orin-backend:
    image: orin-backend:local
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - DB_HOST=mysql
      - DB_PASSWORD=${DB_PASSWORD}
      - JWT_SECRET=${JWT_SECRET}
    env_file: [.env.prod]
    depends_on: [mysql, redis]
```

K8s 部署使用 `ConfigMap` + `Secret` 注入环境变量，不要把密钥写入镜像。
