# ORIN 开发规范

面向贡献者：协作约束、命名规范、提交流程、联调要点。

## 1. 不可违反的架构约束

### 1.1 三层职责不串边

- **orin-backend** 是业务实体唯一持久化方，AI 引擎不直连业务数据库
- **orin-ai-engine** 仅承载工作流/协作的执行内核，不写业务库
- **orin-frontend** 通过 Java 后端访问数据，禁止直连 AI 引擎或外部 provider

### 1.2 协作执行链唯一

- 编排层：`app.engine.collaboration_langgraph`
- 分发消费层：`app.engine.mq_worker`
- 执行内核：`app.engine.task_runtime.TaskRuntime`

**禁止**：在 `mq_worker` 新增独立执行逻辑；在 LangGraph 节点新增第二套执行引擎；新增与 `TaskRuntime` 并行的第三套内核。

### 1.3 接口前缀不混用

| 前缀 | 用途 |
|------|------|
| `/v1/*` | OpenAI 兼容网关（API Key） |
| `/api/v1/*` | 内部业务接口（JWT） |
| `/api/workflows/*` | 工作流管理 |
| `/api/traces/*` | Trace 查询 |

新增模块前先确认是否能复用现有前缀，避免再造一套。

### 1.4 必备字段

- 所有执行型请求携带 `traceId`
- 所有任务对象使用统一状态枚举：`PENDING / RUNNING / COMPLETED / FAILED / CANCELLED`
- 协作包：`PLANNING → DECOMPOSING → EXECUTING → CONSENSUS → COMPLETED / FAILED / FALLBACK`

## 2. 编码规范

### 2.1 Java（后端）

- 包结构：`com.adlin.orin.modules.<模块名>`，模块内分 `controller / service / repository / entity / dto`
- 控制器只做参数校验与返回值组装，业务逻辑下沉到 service
- 实体使用 JPA / MyBatis 任一，避免同模块内混用
- 异常用 `BusinessException(errorCode, message)`，对外统一错误响应格式
- 日志用 SLF4J，敏感字段（密码、token、api key）禁止打印

### 2.2 Python（AI 引擎）

- 类型注解必填，公共函数补 docstring
- 节点处理器统一继承 `BaseNodeHandler`，禁止在 handler 中持久化业务数据
- 长任务必须支持取消信号（`asyncio.CancelledError`）
- 依赖通过 `pyproject.toml` 管理，不维护 `requirements.txt`

### 2.3 前端（Vue 3）

- 组合式 API（`<script setup>`），禁用 Options API 写新代码
- 状态管理用 Pinia，组件间通信优先 props/emit，跨页才用 store
- 路由常量集中在 `router/routes.js`，菜单暴露见 `topMenuConfig.js`
- 同一能力存在多个路径别名时，必须在 `topMenuConfig.js` 标注主入口
- API 请求统一走 `src/api/*`，组件不直接调 `axios`

## 3. 提交规范

### 3.1 Commit Message

格式：`<type>(<scope>): <subject>`

| type | 用途 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| refactor | 重构（不改外部行为） |
| docs | 文档 |
| test | 测试 |
| chore | 构建、依赖、CI |
| perf | 性能优化 |

示例：`feat(workflow): add DSL validator for loop nodes`

### 3.2 PR 必检项

- [ ] 协作执行是否仍经由 `TaskRuntime`，未引入并行执行内核
- [ ] 新增接口已纳入对应前缀分组，未自创新前缀
- [ ] 涉及前端入口的，已更新 `routes.js` 与 `topMenuConfig.js`
- [ ] 涉及环境变量的，已同步 [部署指南.md](./部署指南.md)
- [ ] 单元测试覆盖关键路径
- [ ] 不打印敏感字段、不提交 `.env`

## 4. 联调指南

### 4.1 接口是否真存在

```bash
rg "@RequestMapping|@GetMapping|@PostMapping" \
   orin-backend/src/main/java/com/adlin/orin
open http://localhost:8080/swagger-ui/index.html
```

### 4.2 工作流/协作是否真执行

涉及工作流、协作时**同时**核对：

1. 前端节点配置和入口页
2. Java 控制器与代理层（`WorkflowController` / `CollaborationController`）
3. Python `app/engine` 节点处理器与 TaskRuntime

页面可点开 ≠ 能力闭环。

### 4.3 路由是否仍在使用

```bash
sed -n '1,260p' orin-frontend/src/router/routes.js
sed -n '1,260p' orin-frontend/src/router/topMenuConfig.js
rg "开发中|未启用|待实现" orin-frontend/src
```

### 4.4 鉴权放行

```bash
rg "swagger-ui|v3/api-docs|requestMatchers" \
   orin-backend/src/main/java/com/adlin/orin/security
```

## 5. 测试

```bash
# 后端单测
cd orin-backend && mvn test

# AI 引擎
cd orin-ai-engine && venv/bin/pytest

# 前端单测
cd orin-frontend && npm run test:unit

# 全链路烟测
bash scripts/smoke-test.sh
```

测试必须使用真实组件而非过度 mock，至少在以下情况坚持真实：

- 数据库（用 H2 或 testcontainers）
- 状态机流转
- 鉴权与限流

## 6. 文档维护

- 修改架构、协议、接口前缀时，**同步**更新 [架构设计.md](./架构设计.md) 与 [API文档.md](./API文档.md)
- 模块成熟度变化时，**同步**更新 [功能完成度.md](./功能完成度.md)
- 新增环境变量时，**同步**更新 [部署指南.md](./部署指南.md)
- 文档与代码冲突时，优先修正文档；无法保证准确则删除

## 7. 禁止事项

- 直接在生产 / 主分支推送，必须经 PR
- 提交 `.env`、API Key、密码、私钥
- 把页面入口存在直接当作"能力已完成"
- 为绕过测试而 `--skip-tests` 或注释 `@Test`
- 在 service 中调用其他 service 的私有方法（用接口/事件解耦）
