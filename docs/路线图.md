# ORIN 路线图 · 走向开源项目平均水平

目标：从当前"毕设级骨架"演进到 GitHub 上一个**像样的开源项目**——可被陌生人 clone、跑起来、贡献代码、放心使用。

不追求 SaaS 商用或企业级 SLA，而是对齐 **优秀开源项目的工程基线**：CI 绿、测试有、文档全、社区可参与、版本有节奏。

## 0. 当前状态

- 完成度：骨架 ~80% / 真实闭环 ~55% / 稳定交付 ~50%
- 测试覆盖率：未测量（推测 < 30%）
- CI/CD：无
- 容器化：Dockerfile 与 docker-compose 文件已存在、`config` 静态校验通过，待真实 `docker compose up` smoke 验证
- 社区文件：`CHANGELOG.md` 已建立；仍缺 CONTRIBUTING、ISSUE_TEMPLATE
- 版本管理：无 tag、无 release

## 1. 总体节奏

按单人业余开发节奏，约 **4–5 个月** 可达到目标（含 Phase 0.5 抢热点窗口）。每阶段都有明确的"完成判定"（exit criteria），不达标不进下一阶段。

| 阶段 | 工期 | 主题 | 关键产出 |
|------|------|------|----------|
| Phase 0 | 1–2 周 | 修地基 | CI、Docker Compose、测试基线 |
| **Phase 0.5** ⭐ | **2–3 周** | **MCP 协议接入（差异化关键）** | **MCP Server / Client 双向、Agent Skills、热点曝光** |
| Phase 1 | 4–6 周 | 核心闭环 | P0 链路端到端可跑 |
| Phase 1.5 | 2–3 周 | 角色化体验 | 多角色权限、差异化视图、角色级文档 |
| Phase 2 | 3–4 周 | 质量与可观测 | 覆盖率 60%、统一日志、OTel |
| Phase 3 | 2–3 周 | 安全与运维 | 依赖扫描、备份、审计 |
| Phase 4 | 2–3 周 | 社区化 | 版本、CHANGELOG、贡献文档 |

> **为什么 Phase 0.5 抢在 Phase 1 之前？**
>
> 截至 2026 年 5 月，MCP 已突破 9700 万安装量，被 Linux Foundation 接管治理，成为事实上的"Agent 工具协议"行业标准。Anthropic / OpenAI / Google / Microsoft / ServiceNow 全部跟进，但**国内开源项目接入率仍低**。这是为期 3–6 个月的先发窗口期——窗口关闭后，MCP 接入就从"差异化亮点"变成"基础操作"。
>
> 核心闭环（Phase 1）是必须做的内功，但需要 4–6 周才能见效；MCP 接入只需 2–3 周，能立刻让 ORIN 在 README 和社交媒体上获得辨识度。**先抢热点再练内功**，不抢就错过。

- 当前现象：`nodes.py:117` 在 `completed_subtasks` 满时直接进入 `consensus`，`critic` 判 `FALLBACK` 后路由回 `delegate`，但不会真正重置或重派子任务。

- 后续决策：重派给同一 agent 还是换 agent、是否重置子任务状态、重派次数上限、重派前是否需要人工确认。关键入口见 `nodes.py:458`（critic 计数）、`nodes.py:559`（critic 路由）。

## 8. 关键决策点

走到一半可能遇到的选择题：

| 决策点 | 选项 A | 选项 B | 建议 |
|--------|--------|--------|------|
| 容器编排 | docker-compose | K8s + Helm | A 起步，规模上来再上 B |
| 配置中心 | 环境变量 | Nacos/Consul | A 够用，多环境再考虑 B |
| 消息队列 | RabbitMQ | Kafka | A 够用，吞吐瓶颈再换 B |
| 向量库 | Milvus | Qdrant / pgvector | 现状保持 |
| 模型网关 | 自建 `/v1/*` | LiteLLM | 自建已成型，不必换 |
| 监控 | Prometheus + Grafana | DataDog 类 SaaS | A，开源项目不依赖闭源 SaaS |

**核心原则**：能用免费 / 开源 / 自托管方案的都用 A 方案。开源项目用户最反感的就是依赖一堆付费 SaaS。

## 10. 实际操作建议

- **每个 Phase 开一个 GitHub Milestone**，把 todos 拆成 issue 挂上
- **不要并行多个 Phase**，按顺序走，每完成一个 Phase 发一次小版本（v0.1, v0.2, v0.3, v0.4）
- **不熟的领域不自己写**：CI 模板抄成熟项目、Dockerfile 抄官方示例
- **被卡住时降低标准**：覆盖率 60% 卡住就接受 50%，先继续往下走，回头再补
- **公开记录**：把进展发到知乎/掘金/小红书，外部反馈是最好的动力
- **不要追求完美**：开源项目的优势是"持续迭代"而非"一次到位"
