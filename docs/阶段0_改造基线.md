# ORIN 阶段 0 改造基线

本文档用于收敛 ORIN 当前实现、原始设计方案与本轮改造目标之间的关系，作为后续阶段开发的统一基线。

## 1. 基线结论

- 当前项目的真实实现架构是 `Spring Boot + Vue 3 + Python AI Engine`。
- 原始设计方案 `智能体管理系统设计方案.docx` 提供了需求来源与目标蓝图，但它不在当前仓库中，也不是当前代码的真实技术实现。
- 本轮改造不按“回退到 Yuxi-Know/FastAPI 单体实现”执行，而是在现有 ORIN 架构上补齐核心能力闭环。

## 2. 当前真实架构

### 2.1 系统分层

- `orin-backend`
  - Spring Boot 后端，负责智能体管理、知识库管理、工作流管理、监控、任务、权限、审计、统一 统一网关等。
- `orin-frontend`
  - Vue 3 管理端，负责智能体、知识中心、监控运维、系统管理等页面。
- `orin-ai-engine`
  - 独立 Python FastAPI 执行引擎，负责部分工作流/DSL 执行。

### 2.2 当前主链路

- Agent 调用链
  - 入口：统一 统一网关 `/v1/*` 或智能体管理接口 `/api/v1/agents/*`
  - 核心职责：智能体路由、模型调用、审计、Trace ID 透传
- Workflow 执行链
  - 入口：`/api/workflows/*` 与 `/api/v1/workflow/run`
  - 核心职责：工作流定义管理、实例创建、执行代理到 Python AI Engine、执行审计
- Collaboration 协作链
  - 入口：`/api/v1/collaboration/*`
  - 核心职责：任务包创建、子任务分解、状态推进、事件记录、回退
  - 当前状态：编排骨架已存在，真实执行闭环未完成

## 3. 需求对照基线

| 设计项 | 当前实现状态 | 本轮改造目标 |
| --- | --- | --- |
| 智能体接入与生命周期管理 | 已实现主流程，支持多厂商接入、聊天、版本管理、批量导入导出 | 保持现状，只补链路追踪和协作联动 |
| 知识库与文档管理 | 已实现主流程，含上传、解析、分块、向量化、检索测试、端侧同步接口 | 统一状态机与同步治理 |
| 工作流编排与执行 | 已实现管理与执行基础能力，存在 Java 管理端 + Python 执行引擎双层结构 | 保持双引擎结构，统一 trace、任务、审计口径 |
| 多智能体协作 | 已有协作任务、任务包、子任务、事件总线、前端页面骨架 | 补齐真实执行、重试、回退、事件流闭环 |
| 运行监控 | 已有 Prometheus/硬件监控、审计统计、Trace 查询；Langfuse 有封装但未成为主链路 | 形成“Prometheus + Langfuse + 本地 Trace”三层监控体系 |
| 远程访问网关与统一访问控制 | 已实现统一 统一网关、JWT、限流、审计基础能力 | 统一用户、Trace ID、权限校验链路 |
| 外部框架集成 | Dify、RAGFlow 等已有较多入口与配置页面；部分模块更接近预留位 | 收口为统一接入模型，清理伪完成状态 |
| 端侧知识库同步 | 已实现变更查询、导出、下载、Webhook、checkpoint 接口 | 增加权限、失败重放、同步状态治理 |
| 消息中心、帮助中心、统计分析 | 已有模块和页面，但成熟度不一致 | 从“有页面”升级为“真实业务闭环” |

## 4. 本轮改造范围

### 4.1 纳入本轮

- 多智能体协作闭环
- 任务与队列体系收敛
- 监控主链路统一
- 端侧同步治理
- 外部集成能力收口
- 辅助模块从静态/半成品升级为真实数据驱动

### 4.2 不纳入本轮

- 以 Yuxi-Know 为核心重新搭建后端
- 将 Spring Boot 全量改写为 FastAPI
- 重做现有知识库、工作流、网关成熟功能

## 5. 统一实施约束

### 5.1 架构约束

- Java 后端继续作为系统主控层。
- Python AI Engine 继续作为工作流/DSL 执行引擎，不做移除。
- 新能力优先在现有模块上闭环，不新增平行实现。

### 5.2 数据与链路约束

- 所有执行型请求必须具备统一 `traceId`。
- 所有任务类对象必须统一状态语义，避免 `TaskQueue`、`TaskEntity`、`CollaborationPackage` 各自定义不一致。
- 协作、工作流、智能体调用三条链路都必须可追踪到用户、请求入口、执行结果和失败原因。

### 5.3 文档约束

- 后续开发文档统一以本文档为基线。
- 原始设计文档继续作为“需求来源”，不再作为“当前实现说明”。
