# ORIN 架构设计

## 1. 总体架构

ORIN 采用三层结构，按职责分离：

```text
┌──────────────────────────────────────────────────────────┐
│  orin-frontend (Vue 3 + Element Plus)                    │
│  管理台 · 智能体控制台 · 工作流可视化 · 监控运维         │
└──────────────────────────────────────────────────────────┘
                          │ HTTP / WebSocket
                          ▼
┌──────────────────────────────────────────────────────────┐
│  orin-backend (Spring Boot 3.2)                          │
│  ├─ 业务管控：用户、智能体、知识库、工作流定义、协作     │
│  ├─ 统一网关：/v1/* OpenAI 兼容入口（鉴权、限流、审计）  │
│  └─ 执行代理：把工作流/协作转发到 AI Engine              │
└──────────────────────────────────────────────────────────┘
        │ JDBC                │ HTTP
        ▼                     ▼
   MySQL · Redis      ┌────────────────────────────────────┐
   Milvus              │  orin-ai-engine (FastAPI · Py3.11) │
                       │  ├─ 工作流执行 (DSL Runner)         │
                       │  ├─ 协作编排 (LangGraph)           │
                       │  └─ 任务运行内核 (TaskRuntime)     │
                       └────────────────────────────────────┘
```

## 2. 模块边界

### 2.1 orin-backend（Java 主控层）

- 业务实体的唯一持久化方（MySQL）
- 用户认证、API Key、权限、审计、Trace ID 注入
- 统一网关 `/v1/*`：对外提供 OpenAI 兼容接口，路由到不同 provider
- 工作流管理 `/api/workflows/*`：DSL 定义、版本、实例查询
- 协作管理 `/api/v1/collaboration/*`：任务包、子任务、事件日志

### 2.2 orin-ai-engine（Python 执行层）

- 工作流执行：解析 DSL → 节点处理器 → 真实调用
- 协作执行：LangGraph 编排状态机 → MQ Worker 分发 → TaskRuntime 执行
- 不持久化业务数据，结果回写经由后端

### 2.3 orin-frontend（Vue 管理台）

- 路由结构以 `orin-frontend/src/router/routes.js` 为准
- 顶部菜单 `topMenuConfig.js` 标记每项的发布状态（部分仍为 placeholder）
- 同一能力可能存在多个路由别名（兼容历史路径），不要据此判断"多个能力"

## 3. 三条主链路

### 3.1 Agent 调用链

```text
用户/外部 → /v1/chat/completions（API Key）
         → 网关鉴权 + 限流
         → AgentRouter 路由到 provider
         → 审计落库 + Trace 透传
```

或经管理接口 `/api/v1/agents/{id}/chat`（JWT），用于内部页面调用。

### 3.2 Workflow 执行链

```text
前端编辑 → POST /api/workflows（保存 DSL）
        → POST /api/v1/workflow/run（触发执行）
        → 后端代理 → AI Engine /workflow/execute
        → DSL Runner 调度节点 → 结果回写
```

### 3.3 Collaboration 协作链

```text
前端创建任务包 → POST /api/v1/collaboration/packages
              → 后端入库 + 触发分解
              → AI Engine collaboration_langgraph 编排
              → MQ Worker 分发子任务
              → TaskRuntime 执行（Agent / Workflow / 人工）
              → 事件日志回写
```

## 4. 协作执行架构约束

为防止"LangGraph / MQ Worker / Executor 并存导致执行语义分裂"，协作链路约束如下：

**唯一执行链**：

1. 编排层：`app.engine.collaboration_langgraph` — 状态机、路由、重试决策
2. 分发消费层：`app.engine.mq_worker` — MQ 消费、幂等、结果回写
3. 执行内核：`app.engine.task_runtime.TaskRuntime` — 单任务执行（唯一计算内核）

**禁止事项**：

- 在 `mq_worker` 中新增独立执行逻辑
- 在 LangGraph 节点中新增第二套执行引擎
- 新增与 `TaskRuntime` 并行的第三套执行内核

**兼容**：`collaboration_executor.py` 仅保留兼容导入，`execute_collaboration` 已废弃，统一使用 `run_collaboration`。

## 5. 数据与链路约束

- 所有执行型请求必须携带统一 `traceId`
- 任务类对象统一状态语义：`PENDING → RUNNING → COMPLETED / FAILED / CANCELLED`
- 协作包状态机：`PLANNING → DECOMPOSING → EXECUTING → CONSENSUS → COMPLETED / FAILED / FALLBACK`
- 三条主链路均可追踪到：用户 · 入口 · 执行结果 · 失败原因

## 6. 接口前缀约定

| 前缀 | 用途 | 鉴权 |
|------|------|------|
| `/v1/*` | OpenAI 兼容统一网关 | API Key |
| `/api/v1/*` | 内部业务接口（多数模块） | JWT |
| `/api/workflows/*` | 工作流管理 | JWT |
| `/api/traces/*` | Trace 查询 | JWT |
| `/api/system/*` | 系统集成 / MCP | JWT |

健康检查并存两路径（兼容历史脚本）：

- 后端：`/v1/health`、`/api/v1/health`
- AI 引擎：`/health`、`/v1/health`

## 7. 设计原则

- **页面存在 ≠ 能力闭环**：判断能力可用性必须同时核对前端 + Java 控制器 + Python 执行器
- **不新增平行实现**：新能力优先在现有模块内闭环
- **配置而非代码**：provider、模型、限流等通过配置或管理界面调整，避免硬编码
