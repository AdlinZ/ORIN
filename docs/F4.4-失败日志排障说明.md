# F4.4: 失败日志排障说明

## 概述

本文档说明测试/构建失败时如何快速定位问题。先查具体错误类型，再按对应路径排查。

---

## 一、后端测试失败

### 1.1 编译失败

**现象**：`mvn compile` 报错

**排查路径**：
1. 看报错文件的 import 是否缺失（通常最后一行是真正缺失的类）
2. 如果是 `package does not exist`，检查是否在 `pom.xml` 中添加了新依赖但未 `mvn install`
3. 如果是版本不兼容，检查 Java 版本（需要 Java 17）

```bash
# 清理后重试
mvn clean compile
java -version  # 确认 Java 17
```

### 1.2 测试失败

**现象**：`mvn test` 显示 `Failures: N, Errors: N`

**排查路径**：

```
1. 先看是哪个测试类失败
   → 测试报告在 target/surefire-reports/

2. 查看具体失败信息
   → target/surefire-reports/<TestClass>.txt
   → 包含: 期望值 vs 实际值、完整堆栈

3. 常见失败类型:

a) NullPointerException 在 @BeforeEach
   → mock 对象未正确注入，检查 @Mock / @InjectMocks

b) AssertionError: expected vs actual
   → 业务逻辑与预期不符，检查测试的 given/when 条件

c) IllegalStateException: ApplicationContext
   → @SpringBootTest 加载失败，检查 @ActiveProfiles("test")
     和 test/resources/application-test.properties

d) H2 Schema 冲突 (Index already exists)
   → Flyway 和 Hibernate ddl-auto 冲突
   → 确认 spring.flyway.enabled=false

e) Milvus/Redis Connection refused
   → 这是外部依赖，确认 test profile 是否 mock 了这些服务
```

### 1.3 特定测试类单独运行

```bash
# 只运行某个测试类（快速定位）
mvn test -Dtest=GlobalExceptionHandlerTest

# 只运行失败的测试方法
mvn test -Dtest=GlobalExceptionHandlerTest#testHandleBusinessException

# 完整输出模式（看到更多日志）
mvn test -Dtest=GlobalExceptionHandlerTest -X
```

### 1.4 TraceId 在日志中定位

所有后端日志都会通过 `MDC` 携带 `traceId`。当测试失败时：

1. 在测试报告的堆栈中找到 `traceId`（格式：`[TraceId: xxxxxx]`）
2. 在日志文件中搜索该 traceId：
   ```bash
   grep "TraceId: <your-trace-id>" target/logs/application.log
   ```
3. 同一 traceId 下的所有日志代表同一次请求的完整调用链

## 二、前端 Lint 失败

**现象**：`npm run lint` 报错

**排查路径**：

```
1. 直接看终端输出（会显示具体文件和行号）

2. 常见错误类型:

a) 'xxx' is defined but never used
   → 删除未使用的 import 或变量

b) Component name "xxx" should use
   → Vue 组件命名不符合规范（PascalCase）

c) Unexpected trailing comma / semicolon
   → 运行 npm run lint:fix 自动修复

d) Maximum call stack size exceeded
   → 可能是循环依赖，检查 import 图

3. 自动修复格式问题:
   npm run lint:fix
```

## 三、前端构建失败

**现象**：`npm run build` 报错

**排查路径**：

```
1. 看终端输出的最后 5-10 行（通常包含真正错误）

2. 常见原因:

a) Module not found: 'xxx'
   → node_modules 未正确安装，运行 npm install

b) SyntaxError: /path/file.vue
   → .vue 文件中有语法错误，检查 <script> 和 <template>

c) Resource: /xxx not found
   → 静态资源路径问题，检查 vite.config.js 的 publicDir

d) CJS / ESM 混淆
   → 检查 package.json 的 type 字段和依赖的 module 类型

3. 构建产物验证:
   npm run build 后检查 dist/ 目录是否生成
```

## 四、AI 引擎测试失败

**现象**：`python3 -m pytest tests/ -v` 报错

**排查路径**：

```
1. 看终端输出的 FAILED 行（显示具体哪个测试）

2. 常见失败:

a) ImportError: No module named 'xxx'
   → 依赖未安装: pip install -r requirements.txt

b) ConnectionError: HTTPConnectionPool(host='localhost', port=8000)
   → 被测服务未启动，需要先启动 FastAPI 服务

c) AssertionError: expected 'xxx' != actual 'yyy'
   → 业务逻辑与预期不符，检查测试用例的假设

d) Failed to load OpenAPI spec
   → app/main.py 启动失败，先手动 python3 app/main.py 看报错

3. 单独运行失败测试:
   python3 -m pytest tests/test_llm_handler.py::test_chat -v
```

## 五、日志链路追踪

### traceId 在 CI 中的传递

```
请求 → TraceIdFilter (生成/读取 traceId，写入 MDC)
     ↓
Controller → Service → Repository (同一 traceId)
     ↓
响应头: X-Trace-Id: <traceId>
     ↓
日志文件搜索: grep "TraceId: <traceId>" logs/app.log
```

### 日志级别说明

| 级别 | 使用场景 |
|------|---------|
| ERROR | 异常被捕获、明确失败（如 AI 调用返回错误） |
| WARN | 降级行为（如 OCR 服务未配置，使用备选方案） |
| INFO | 关键业务节点（开始/结束一次工作流、agent 切换） |
| DEBUG | 详细执行路径（SQL 语句、参数值） |

**当测试失败时**：
- ERROR 级别日志通常包含根本原因
- 搜索 `ERROR` 和对应 traceId 即可定位
