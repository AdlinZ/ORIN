# 原始设计与当前实现对照

这份文档的目的不是证明“已经实现了多少”，而是帮助大家区分早期设计目标和当前仓库真实落地之间的差异，避免继续拿旧方案直接套现状。

## 基本结论

- 原始设计更接近“以 Yuxi-Know / FastAPI 为核心基座的单体或准单体方案”
- 当前仓库已经演化成 “Spring Boot 主后端 + Vue 3 前端 + Python AI Engine” 的双后端结构
- 因此很多设计概念仍然有参考价值，但不能直接等同于当前已实现能力

## 架构对照

| 维度 | 原始设计倾向 | 当前实现 | 说明 |
| --- | --- | --- | --- |
| 后端主架构 | FastAPI 为主 | Spring Boot 为主，FastAPI 负责 AI 执行 | 已偏离原路线 |
| 前端 | 管理端后台 | Vue 3 + Element Plus | 已形成独立管理台 |
| 数据存储 | 单体知识平台思路 | MySQL + 向量库 + 外部集成 | 更工程化 |
| 执行方式 | 平台内一体化执行 | Java 管理 + Python 执行器 | 双链路协同 |
| 接口风格 | 偏统一平台接口 | 业务接口 + OpenAI 兼容网关并存 | 需文档收口 |

## 能力对照

| 能力 | 原始目标 | 当前情况 | 判断 |
| --- | --- | --- | --- |
| 智能体管理 | 接入、配置、调用、监控 | 已有明确模块和前后端入口 | 已成型 |
| 知识库 | 上传、解析、检索、图谱、同步 | 基础链路已在，图谱和多源治理仍在收敛 | 部分达成 |
| 工作流 | DSL 编排与执行 | 编辑器和执行器都存在，但复杂节点要逐项确认 | 部分达成 |
| 多智能体协作 | 任务分解、并行、评审、共识 | 架构和页面较完整，成熟度仍偏低 | 偏早期 |
| 统一观测 | 一体化 tracing / audit / metrics | 多套能力并存，统一口径仍待收敛 | 部分达成 |
| 外部集成 | Dify、RAGFlow 等统一纳管 | 已有多个入口，但治理模型仍在完善 | 部分达成 |

## 最容易混淆的地方

### 1. OpenAI 网关和业务接口并存

当前仓库同时存在：

- `/v1/*`：偏 OpenAI 兼容网关
- `/api/v1/agents/*`：偏平台业务接口
- `/api/workflows/*`、`/api/v1/workflow/*`：偏工作流管理与执行

所以旧设计里“统一调用入口”的说法，在当前实现中需要拆开理解。

### 2. 页面完整度不等于能力闭环

当前前端很多页面已经比较完整，但以下能力仍要谨慎判断：

- 多智能体协作
- 知识图谱
- 部分多模态链路
- 外部系统同步治理

### 3. 双后端意味着验收要看两边

如果一个能力涉及工作流、协作、AI 执行，不仅要看 Java 控制器，还要同时检查：

1. 前端入口
2. Spring Boot 代理层
3. Python AI 引擎 handler / executor

## 当前应采用的基线

从今天开始，所有设计对齐、任务拆分和文档评审，建议统一以以下基线为准：

- 当前代码结构
- 当前路由与控制器
- 当前测试和脚本能否跑通
- [系统功能实现评估报告.md](./系统功能实现评估报告.md) 中的保守口径

如果需要引用原始设计，应把它视为“目标蓝图”，而不是“已完成功能清单”。
