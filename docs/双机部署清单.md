# ORIN 双机部署清单

本文档说明 ORIN 在“两台服务器”场景下的推荐部署方式。

当前前提：

- 云服务器已部署 `MySQL`
- 云服务器承载应用入口和核心应用
- 本地服务器承载大部分外部服务与观测组件

本文档以当前 ORIN 仓库真实依赖为准，不完全等同于早期的 Yuxi-Know 设计方案。

## 一、部署前提

### 云服务器

建议放置以下组件：

- `orin-frontend`
- `orin-backend`
- `orin-ai-engine`
- `MySQL`
- `Nginx`

### 本地服务器

建议放置以下组件：

- 向量检索、消息队列、观测、日志等外部服务
- 后续补齐的图谱、对象存储、日志聚合等组件

## 二、外部服务部署建议表

| 服务 | 是否必需 | 推荐部署位置 | 作用 | 默认端口 | 访问方向 | 是否允许公网暴露 | 备注 |
|------|------|------|------|------|------|------|------|
| `MySQL` | 必需 | 云服务器 | 业务主库 | `3306` | 云后端 -> MySQL | 否 | 当前 ORIN 真实实现使用 MySQL，不是 PostgreSQL |
| `Redis` | 必需 | 优先云服务器 | 限流、缓存、共享状态 | `6379` | 云后端 -> Redis | 否 | 最好与 `orin-backend` 同侧，避免跨公网延迟 |
| `Milvus` | 知识库检索时必需 | 本地服务器 | 向量检索 | `19530` | 云后端 -> 本地 Milvus | 否 | 只要要用知识库检索，基本应视为必需 |
| `Nginx` | 必需 | 云服务器 | 统一入口、反向代理、TLS 终止 | `80/443` | 用户 -> Nginx | 是 | 仅暴露入口层，后端服务不直接暴露 |
| `RabbitMQ` | 按需 | 优先云服务器，也可本地 | 异步任务队列、重试、优先级调度 | `5672/15672` | 云后端 -> RabbitMQ | 否 | 管理端口不应暴露公网；跨公网时稳定性会下降 |
| `Langfuse` | 按需 | 本地服务器 | LLM 链路追踪与成本分析 | 常见 `3000` | 云后端 -> 本地 Langfuse | 不建议 | 建议通过受控入口或内网访问 |
| `Prometheus` | 按需 | 本地服务器 | 系统与基础设施监控 | `9090` | Grafana/云后端 -> Prometheus | 不建议 | 建议只开放内网或 VPN |
| `Grafana` | 按需 | 本地服务器 | 监控展示面板 | 常见 `3000` | 运维人员 -> Grafana | 可受控暴露 | 如需外网访问，应加认证与白名单 |
| `Neo4j` | 按完整方案补齐 | 本地服务器 | 知识图谱存储 | `7474/7687` | 云后端 -> 本地 Neo4j | 否 | 当前仓库图谱能力未完全闭环，后续再补更稳妥 |
| `MinIO` | 按完整方案补齐 | 本地服务器 | 对象存储、归档 | `9000/9001` | 云后端 -> 本地 MinIO | 否 | 建议只开放给应用层访问 |
| `ELK` | 按完整方案补齐 | 本地服务器 | 日志聚合与检索 | 多端口 | 各服务 -> ELK | 否 | 仅建议内网使用 |
| `Dify` | 按需 | 通常本地服务器 | 外部智能体平台接入 | 依部署而定 | 云后端 -> Dify | 不建议 | 仅在需要纳管 Dify 时部署 |
| `RAGFlow` | 按需 | 通常本地服务器 | 外部知识库平台接入 | 常见 `9380` | 云后端 -> RAGFlow | 不建议 | 仅在需要同步 RAGFlow 时部署 |
| `Ollama` | 按需 | 通常本地服务器 | 本地模型推理服务 | 常见 `11434` | 云后端 -> Ollama | 不建议 | 适合本地自托管模型场景 |
| `vLLM` | 按需 | 通常本地服务器 | 本地模型推理服务 | 依部署而定 | 云后端 -> vLLM | 不建议 | 适合统一 OpenAI 兼容接口场景 |

## 三、推荐落位

### 云服务器建议放置

- `orin-frontend`
- `orin-backend`
- `orin-ai-engine`
- `MySQL`
- `Nginx`
- `Redis`
- 可选 `RabbitMQ`

### 本地服务器建议放置

- `Milvus`
- `Langfuse`
- `Prometheus`
- `Grafana`
- 后续 `Neo4j`
- 后续 `MinIO`
- 后续 `ELK`

## 四、特别说明

### 1. Redis 最好与云后端同侧

原因：

- 当前 Redis 不只是缓存，还承担限流与共享状态
- 如果 Redis 放在本地服务器，云后端每次访问都会跨公网
- 这会直接放大接口延迟，并影响限流和状态一致性

结论：

- `Redis` 优先放云服务器

### 2. RabbitMQ 也优先与云后端同侧

原因：

- RabbitMQ 涉及消息投递、消费确认、失败重试
- 如果跨公网连接，任务调度可靠性和吞吐都会变差

结论：

- `RabbitMQ` 最好也放云服务器
- 如果必须放本地，应通过 VPN、专线或其他稳定内网通道接入

### 3. Milvus 放本地通常可以接受

原因：

- Milvus 更多承担向量检索能力
- 相比 Redis 和 RabbitMQ，它对低延迟的敏感度通常略低

前提：

- 云后端到本地服务器链路稳定
- 访问具备认证和网络隔离

## 五、网络与安全要求

推荐云服务器到本地服务器之间使用以下方式之一：

- `VPN`
- `Tailscale`
- `ZeroTier`
- 专线或内网打通

### 严禁直接公网暴露的组件

- `Redis`
- `RabbitMQ`
- `Milvus`
- `Neo4j`
- `MinIO`

### 安全要求

- 仅允许云服务器固定 IP 访问本地服务
- 所有中间件启用认证密码、Token 或 Access Key
- 尽量使用 TLS、隧道或内网链路，不走明文公网连接
- 管理端口仅对内网或受控运维入口开放
- 不要把中间件管理界面直接挂到公网

## 六、最小可落地方案

### 第一阶段

先部署最小闭环：

- `MySQL`
- `Redis`
- `Milvus`
- `Nginx`

这一步可以先满足：

- 应用运行
- 用户访问
- 基础知识库检索
- 基础接口限流与缓存

### 第二阶段

补齐异步与观测：

- `RabbitMQ`
- `Langfuse`
- `Prometheus`
- `Grafana`

这一步可以增强：

- 异步任务处理
- LLM 调用追踪
- 系统与中间件监控

### 第三阶段

补齐完整平台能力：

- `Neo4j`
- `MinIO`
- `ELK`

这一步主要面向：

- 图谱能力补强
- 对象存储治理
- 日志聚合与检索

## 七、风险提示

- 本地服务器断网，会直接影响云侧访问本地中间件
- 跨公网直连 `Redis` 和 `RabbitMQ` 风险最高
- 监控、日志、图谱、对象存储这类组件适合放本地，但前提是链路稳定
- 当前 ORIN 真实实现基线是 `Spring Boot + MySQL + Redis + Milvus + FastAPI AI Engine`
- 不应完全按原始 Yuxi-Know 方案理解当前 ORIN 的已实现能力

## 八、实施时的核对项

后续如果真正按本文部署，建议至少核对以下内容：

- 组件名称与仓库配置一致
- 端口与环境变量名称与现有文档一致
- 不把“目标方案”误写成“当前仓库已全部落地”
- 云服务器到本地服务器的链路已打通
- 本地中间件未直接暴露公网
- `Redis`、`RabbitMQ`、`Milvus` 已启用认证
