# ORIN 开发 Agent 执行说明

> 面向：Codex、Claude Code、Cursor Agent、内部开发型智能体
> 版本：`2026-03-31`
> 用途：让接手 ORIN 的开发 agent 在最短时间内建立正确上下文，避免继续基于旧设计、旧文档或页面表象做出错误修改。

## 0. 远端协作前提

默认前提是：

- 你运行在另一台机器上
- 你通过 git 拉取 ORIN 仓库
- 你看不到作者本机的绝对路径、桌面文件或本地临时资料

因此你必须遵守：

- 只相信仓库内实际存在的文件
- 文档中的内部引用必须能在当前仓库中打开
- 如果某条说明依赖仓库外文件，应明确视为“额外背景”，不能作为阻塞前提

## 1. 先说结论

你接手的 ORIN，不是一个“已经基本完成、只差零碎修修补补”的项目。

它的真实状态是：

- 平台骨架已经很完整
- 多个模块已经有页面、接口、实体和测试文件
- 但不少高级能力仍处于半闭环、弱实现、占位或多文档口径不一致的状态

你的首要任务不是继续堆功能，而是：

1. 先确认现状
2. 再补真实闭环
3. 再补测试、文档、错误语义和运维边界

## 3. 当前真实架构

不要按原始设计里的 Yuxi-Know / FastAPI 单体思路理解当前系统。

当前真实架构是：

- `orin-backend`
  - Spring Boot 主控后端
  - 负责系统管理、智能体管理、知识库管理、工作流管理、权限、安全、审计、任务、网关等
- `orin-frontend`
  - Vue 3 管理端
  - 负责智能体、知识中心、监控运维、系统管理等入口和交互
- `orin-ai-engine`
  - Python FastAPI 执行引擎
  - 负责部分工作流 DSL、节点处理、协作执行

结论：

- Java 后端是主控层，不要假设 Python 引擎就是系统主后端
- Python 引擎是执行层，不要在 Java 和 Python 之间重复造能力
- 前端页面存在，不等于整条链路已闭环

## 4. 你必须遵守的判断规则

### 4.1 永远优先相信代码，不优先相信旧文档

如果文档与代码冲突，优先级如下：

1. 当前代码
2. [ORIN统一开发总计划.md](./ORIN统一开发总计划.md)
3. 当前评估类文档
4. 原始设计方案
5. 历史文档

### 4.2 页面存在不等于能力完成

以下情况一律不能算“完成”：

- 前端有页面，后端没有闭环
- 控制器有接口，service 仍是弱实现、mock、占位或默认异常
- Java 入口存在，但 Python handler 仍是 placeholder
- 依赖外部服务，默认环境根本跑不通

### 4.3 一旦涉及执行链，就必须看三层

如果任务涉及：

- 工作流
- 协作
- 知识处理
- Agent 执行

你必须同时检查：

1. 前端入口和参数来源
2. Java 后端 controller/service/proxy
3. Python AI engine executor/handler

少看一层，判断大概率会偏。

### 4.4 “有测试文件”不代表“有可靠测试基线”

判断测试成熟度时，要看：

- 能不能直接跑
- 跑通依赖是否明确
- 是单元测试、集成测试还是 mock 测试
- 是否覆盖真实主链路

## 5. 开发优先级

默认优先级必须按这个顺序走：

1. 文档口径与现状统一
2. 工程基线、启动基线、测试基线
3. 智能体、知识库、工作流三条主链闭环
4. 协作闭环与统一观测
5. 知识图谱、多模态、外部集成、端侧同步
6. 驾驶舱、分析、辅助模块和体验优化

除非用户明确要求，否则不要跳过前面的基础工作，直接去做展示型功能。

## 6. 接到任务后的标准工作流

每次开发，统一按以下步骤执行。

### 第一步：定位任务影响范围

至少回答这几个问题：

- 这是哪个模块的问题
- 只影响前端，还是前后端一起影响
- 是否涉及执行链
- 是否涉及权限、安全、配置、任务状态或 trace
- 是否涉及外部依赖

### 第零步：先确认 git 同步状态

由于你与其他机器通过 git 协作，开始前至少确认：

- 当前分支是什么
- `git status` 是否干净
- `git fetch` 后是否落后于远端
- 最近是否有人在同一模块上持续提交

### 第二步：搜当前真实入口

建议最少做这些检查：

```bash
rg -n "相关模块关键词" orin-frontend/src orin-backend/src/main/java orin-ai-engine/app
```

```bash
rg -n "@RequestMapping|@GetMapping|@PostMapping|@PutMapping|@DeleteMapping" orin-backend/src/main/java/com/adlin/orin
```

```bash
rg -n "placeholder|mock|UnsupportedOperationException|TODO|return null" orin-backend orin-ai-engine
```

### 第三步：判断是真缺功能，还是文档/状态不一致

常见情况只有四种：

1. 真没实现
2. 实现了，但页面/接口没接好
3. 实现了，但错误语义、状态机、trace 没收口
4. 实现了，但文档写错了

不要一上来就写代码，先判断是哪一种。

### 第四步：优先补闭环，不优先补表面

优先顺序：

1. 服务真实实现
2. 状态机和错误语义
3. trace / 审计 / 任务状态
4. 前端交互和反馈
5. 文档
6. 体验优化

### 第五步：验证

每次至少做一类验证：

- 单元测试
- 集成测试
- smoke 脚本
- 手动最小闭环验证

如果不能验证，要明确说明为什么不能验证。

### 第六步：同步文档

只要你改变了以下任一项，就要同步文档：

- 功能边界
- 接口行为
- 配置方式
- 启动方式
- 错误语义
- 验收状态

## 6.1 接任务前检查清单

- [ ] `git status` 确认没有未提交改动会干扰当前任务
- [ ] `git fetch` 并确认当前分支没有明显落后
- [ ] 阅读目标模块最近几次提交，确认不是并行冲突热点
- [ ] 确认相关文档在当前仓库中真实存在
- [ ] 确认你即将修改的是当前真实入口，而不是旧路径或兼容路径

## 6.2 修改前必查项

- [ ] 前端是否真的有对应路由或实际挂载页面
- [ ] 后端是否真的有 controller + service 闭环
- [ ] 如果涉及执行链，Python AI engine 是否参与其中
- [ ] 是否已经存在 mock、placeholder、默认异常或页面复用
- [ ] 是否依赖仓库外配置、第三方服务或人工前置数据

## 6.3 修改后最小验证

最少按改动类型完成以下验证：

- 后端改动：至少执行一次 `mvn compile`
- 前端改动：至少执行一次 `npm run build`，若项目约束更轻则至少说明未执行原因
- Python AI 引擎改动：至少执行能覆盖本次修改的最小运行或测试命令
- API 改动：至少用 curl、Swagger 或前端调用验证不是裸 500
- 执行链改动：至少验证 Java 和 Python 两端都没有断链

## 6.4 文档同步触发规则

以下情况必须同步文档：

- 新增、删除、修改了 API 路径或关键参数
- 新增、删除、修改了前端路由或页面入口
- 修改了部署方式、启动方式、依赖方式
- 修改了模块边界、状态判断、错误语义
- 删除了旧文档、旧入口或旧兼容路径

以下情况通常不需要单独改文档：

- 单纯改注释
- 纯样式调整且不影响入口和行为
- 不改变对外行为的内部重构

## 7. 什么算一次合格的修改

只有同时满足下面条件，才算合格：

- 修的是根因，不只是掩盖报错
- 改完后链路更清楚，而不是更绕
- 补了最小验证
- 后端修改至少应能编译通过，前端修改至少应能构建或说明未构建原因
- 没有引入新的伪完成状态
- 文档没有继续误导下一个 agent

## 8. 各模块工作时的特别注意事项

> 注意：下面这些“模块名称”是工作视角，不一定和实际代码目录一一对应。实际路径请以 `orin-backend/src/main/java/com/adlin/orin/modules/`、`orin-frontend/src/`、`orin-ai-engine/app/` 为准。

## 8.1 智能体模块

重点关注：

- 生命周期状态是否统一
- 接入、聊天、导入导出、刷新元数据是否真做实
- 异步任务有没有状态回查
- provider 切换是否影响审计和 trace

不要默认以下能力已经完全可靠：

- 批量导入导出
- 元数据刷新
- 异步任务状态查询
- 多 provider 一致性

## 8.2 知识库模块

重点关注：

- 上传、解析、分块、向量化、检索是否形成统一任务状态机
- OCR/ASR 未配置时是否明确降级
- 检索结果是否真的来自真实链路
- 同步、导出、变更查询是否有失败治理

不要先做：

- 只改页面样式
- 只补图谱入口
- 只补演示数据

先做：

- 处理链闭环
- 失败原因
- 任务状态可追踪

## 8.3 工作流模块

重点关注：

- Java 管理层和 Python 执行层的 DSL 是否一致
- 节点输入输出是否有统一结构
- provider 注入是否会错误 fallback
- 复杂节点是否仍是模拟语义

尤其要谨慎检查：

- `iteration`
- `loop`
- `knowledge_retrieval`
- 条件分支
- 流式输出

## 8.4 协作模块

这是最容易被高估的模块。

重点关注：

- 串行协作是否真实执行
- 并行协作结果是否真实汇总
- 超时、取消、重试、回退是否有明确语义
- shared memory 是否真共享，而不是只在单次上下文里传对象

尤其要检查：

- `delegate`
- `parallel_fork`
- `consensus`
- `critic`
- `memory_read`
- `memory_write`

## 8.5 观测与运维模块

重点关注：

- `traceId` 是否贯穿入口到执行
- Prometheus、Langfuse、Audit/Trace 是否职责清晰
- 告警源是否统一
- 面板数据是否来自真实接口

不要把“有 dashboard”直接写成“统一观测已完成”。

## 8.6 系统与安全模块

重点关注：

- 用户、角色、部门、资源权限是否一致
- 未登录、越权、权限不足响应是否统一
- 配置项是否有 schema、校验、默认值边界
- 敏感操作是否需要二次确认

## 8.7 外部集成与端侧同步

重点关注：

- 统一接入模型是否存在
- 每种集成是否有健康状态、错误状态、最近同步状态
- 同步是否支持 checkpoint、失败重放、冲突处理

不要给每个外部系统单独造一套完全不同的状态逻辑。

## 9. 发现这些情况时，优先修正文档或状态，而不是硬写功能

以下情况很常见：

- 文档写“已完成”，代码里其实是 mock
- 页面有入口，但后端默认异常
- 测试文件存在，但依赖没配
- 设计方案写得很大，但当前架构根本不是那条路线

这种时候，你应该优先：

1. 修正状态口径
2. 记录真实边界
3. 再决定要不要继续补实现

不要为了“看起来完成”继续扩大误差。

## 9.1 禁止事项

- 禁止依赖作者本机绝对路径或仓库外文件作为开发前提
- 禁止在未确认接口存在的情况下直接写前端调用
- 禁止把 mock、placeholder、默认异常路径当成真实交付
- 禁止只改前端或只改后端就默认宣称功能完成
- 禁止跳过验证步骤直接声称“已修复”

## 10. 提交前检查清单

提交前至少确认：

- `git status` 看过，没误覆盖用户改动
- 改动覆盖到了真正生效的入口
- 至少做了一条验证
- 如果改了接口或行为，文档同步了
- 如果改了执行链，三层都核过
- 如果能力仍有边界，明确写出来了

## 11. 给开发 agent 的默认输出要求

每次完成任务后，默认要向使用者说明四件事：

1. 改了什么
2. 为什么这么改
3. 怎么验证
4. 还有什么风险/边界没解决

如果没有跑测试，必须直说。

## 12. 一句版执行口令

先读总计划和基线文档，按当前代码而不是旧设计理解系统；凡是涉及工作流、协作、知识链路都同时检查前端、Java、Python 三层；优先补真闭环、状态机、错误语义、trace 和测试，不要继续制造伪完成状态。
