# ORIN 开发 Agent 执行说明（单人项目 / 两周交付）

> 面向：Codex、Claude Code、Cursor Agent、内部开发型智能体  
> 版本：`2026-04-11`  
> 场景：项目实际只有 1 名开发者（用户本人）+ 1 个 AI Agent，要求 2 周内完成可交差版本。

## 0. 强制上下文（必须先接受）

1. 本项目不是团队并行开发场景，是单人高压交付场景。  
2. 用户负责最终修改与拍板，Agent 负责高强度实现、验证与文档同步。  
3. 目标不是“理想平台全量完成”，而是“两周内可交差”。

## 1. 交付目标（两周内）

Agent 只对以下目标负责，不得发散：

1. 核心链路可复现跑通。  
2. 关键页面可用、无明显阻断报错。  
3. 关键失败场景有可读错误语义。  
4. 每项改动有最小验证证据（命令、结果、日志/截图说明）。

## 2. 范围控制

## 2.1 必做（Must Have）

1. 观测保底
- `/dashboard/runtime/overview` 可稳定出数
- `/dashboard/runtime/traces` 支持 traceId 查询闭环

2. 协作保底
- 任务包：创建 -> 分解 -> 执行 -> 汇总 -> 完成/失败落库
- 至少 2 条验收用例（成功 1 条、失败 1 条）

3. 外部集成保底
- Dify 或 RAGFlow 至少 1 个完成：配置保存 + 连通测试 + 真实调用

4. MCP / 同步保底
- `mcp-service`：增删改查 + 测试连接
- `client-sync`：查询变更 + 触发同步 + 查看结果

5. 最低测试保底
- 前端、后端、AI 引擎各至少 1 条 smoke 可执行

## 2.2 禁止发散（Won’t Have）

1. 禁止大规模架构重构。  
2. 禁止承诺 AutoGen/CrewAI 两周内完整上线（可标预留）。  
3. 禁止优先做展示型/美化型但不形成闭环的工作。  
4. 禁止为“完美方案”牺牲“可交付时间”。

## 3. Agent 默认执行模式

## 3.1 不阻塞原则

除以下 4 类外，Agent 必须默认继续推进，不得反复等待用户确认：

1. 不可逆破坏操作（批量删除、覆盖历史、重置数据）  
2. 安全/权限高风险变更（鉴权策略、权限边界）  
3. 高成本不可回收外部动作（付费调用、大规模生产写入）  
4. 存在多路线且后果差异显著、无法用可逆默认值规避

## 3.2 默认决策顺序

1. 当前可运行代码  
2. `docs/ORIN统一开发总计划.md`  
3. `docs/系统功能实现评估报告.md`  
4. 本执行说明  
5. 历史文档与旧设计

## 4. 接任务后的标准流程（必须按序）

## 第一步：10 分钟快速定类

将问题归类为：

1. 真缺功能  
2. 前后端未接通  
3. 状态机/错误语义/trace 未收口  
4. 文档口径不一致

超过 10 分钟仍不确定时：按“最可能类型 + 可逆方案”先做。

## 第二步：只看真实入口（禁止想当然）

必查路径：

- `orin-frontend/src/router/`
- `orin-frontend/src/views/`
- `orin-backend/src/main/java/com/adlin/orin/modules/`
- `orin-ai-engine/app/engine/`

建议检索：

```bash
rg -n "关键词" orin-frontend/src orin-backend/src/main/java orin-ai-engine/app
rg -n "placeholder|mock|TODO|UnsupportedOperationException|return null" orin-backend orin-ai-engine
```

## 第三步：固定实现优先级

1. 服务真实实现  
2. 状态机与错误语义  
3. trace / 审计 / 任务状态  
4. 前端交互反馈  
5. 文档同步  
6. 体验优化

## 第四步：最小验证（不可省略）

- 后端改动：至少 `mvn compile`  
- 前端改动：至少 `npm run build`（若未执行必须说明原因）  
- AI 引擎改动：至少 1 条最小运行/测试命令  
- API 改动：至少验证非裸 `500`

## 第五步：交付输出

每次任务结束，必须输出：

1. 改了什么  
2. 为什么这样改  
3. 如何验证  
4. 仍存在哪些风险/边界

## 5. 两周排程（供 Agent 拆解日任务）

## Week 1

1. Day 1-2：观测页与 Trace 查询闭环。  
2. Day 3-4：协作最小闭环 + 成功/失败用例。  
3. Day 5：错误语义与状态收口，文档更新。  
4. Day 6-7：回归与缺陷修补缓冲。

## Week 2

1. Day 8-9：Dify 或 RAGFlow 保底闭环。  
2. Day 10-11：MCP + client-sync 最小闭环。  
3. Day 12：三端 smoke 固化。  
4. Day 13：全链路回归，清 P0/P1。  
5. Day 14：整理交付包（能力清单、验证证据、已知边界）。

## 6. 提交前硬检查清单

- [ ] 改动是否命中真实入口  
- [ ] 至少完成一条可执行验证  
- [ ] 是否避免制造新的“伪完成状态”  
- [ ] 接口/配置/行为变化是否同步文档  
- [ ] 如涉及执行链，是否核对前端+Java+Python 三层

## 7. 禁止事项

1. 禁止把页面存在等同于能力完成。  
2. 禁止把 mock/placeholder/默认异常路径当交付能力。  
3. 禁止跳过验证直接声称“修复完成”。  
4. 禁止未确认接口存在就直接写前端调用。  
5. 禁止把“外部依赖不可达”当作完全停工理由（应先完成本地可做部分）。

## 8. 一句版执行口令

在 ORIN 当前代码现实下，按两周可交差目标推进：优先修核心闭环、状态机、错误语义、trace 与最小测试；非高风险默认不停机，持续交付可验证增量并同步文档边界。
