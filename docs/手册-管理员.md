# ORIN 管理员操作手册

本文档面向 ORIN 平台管理员，涵盖日常运维操作、备份恢复、告警处理与灾难恢复。

---

## 1. 日常健康检查

### 1.1 服务状态检查

```bash
# 检查所有容器状态
docker compose ps

# 检查后端健康端点
curl http://localhost:8080/actuator/health

# 检查 AI Engine 健康
curl http://localhost:8000/health

# 检查数据库连接
docker exec orin-mysql mysql -uroot -p"${MYSQL_ROOT_PASSWORD}" -e "SELECT 1" orindb
```

### 1.2 日志检查

```bash
# 查看后端实时日志
tail -f orin-backend/logs/orin-backend.log | grep -E "ERROR|WARN"

# 查看 AI Engine 日志
docker compose logs -f ai-engine

# 查看 JSON 结构日志（搜索 traceId）
grep "traceId.*abc123" orin-backend/logs/orin-backend.json | jq .

# 查看最近的错误
grep "\"level\":\"ERROR\"" orin-backend/logs/orin-backend.json | tail -20
```

### 1.3 资源检查

```bash
# 磁盘空间
df -h

# Docker 容器资源
docker stats --no-stream

# MySQL 连接数
docker exec orin-mysql mysql -uroot -p"${MYSQL_ROOT_PASSWORD}" -e \
  "SHOW STATUS LIKE 'Threads_connected'; SHOW PROCESSLIST;" orindb

# RabbitMQ 队列深度
docker exec orin-rabbitmq rabbitmqctl list_queues -p /
```

## 2. 备份与恢复

### 2.1 备份

```bash
# 完整备份（所有组件）
bash scripts/backup.sh

# 备份后查看保留的份数
ls -la backups/

# 仅备份 MySQL（日常增量）
bash scripts/backup.sh --keep 30  # 保留 30 份
```

### 2.2 恢复

```bash
# 查看可用备份
ls backups/

# 全量恢复
bash scripts/restore.sh backups/orin-20260604-120000 --all

# 仅恢复 MySQL（数据损坏时）
bash scripts/restore.sh backups/orin-20260604-120000 --mysql

# 仅恢复配置（修改配置后回退）
bash scripts/restore.sh backups/orin-20260604-120000 --config
```

### 2.3 定时备份（Cron）

```cron
# 每天凌晨 2:00 执行备份，保留 30 份
0 2 * * * cd /path/to/ORIN && bash scripts/backup.sh --keep 30 >> /var/log/orin-backup.log 2>&1
```

## 3. 常见告警与处理

### 3.1 AI Engine 不可达

**症状**：`AiEngineMcpClient.McpToolCallException` 或 `Connection refused` 错误堆积

**处理步骤**：
```bash
# 1. 检查 AI Engine 进程
docker compose ps ai-engine
docker compose logs ai-engine --tail=50

# 2. 重启 AI Engine
docker compose restart ai-engine

# 3. 验证恢复
curl http://localhost:8000/health

# 4. 检查积压的 Workflow 任务
curl http://localhost:8080/api/v1/tasks?status=QUEUED | jq '.total'
```

### 3.2 数据库连接池耗尽

**症状**：后端 500，错误信息含 `Connection pool exhausted`

**处理步骤**：
```bash
# 1. 检查 MySQL 连接
docker exec orin-mysql mysql -uroot -p"${MYSQL_ROOT_PASSWORD}" -e \
  "SHOW PROCESSLIST; SHOW STATUS LIKE 'Max_used_connections';" orindb

# 2. 如果连接数过多，重启后端释放连接池
docker compose restart backend

# 3. 检查是否有慢查询
docker exec orin-mysql mysql -uroot -p"${MYSQL_ROOT_PASSWORD}" -e \
  "SHOW FULL PROCESSLIST;" orindb | grep -v "Sleep" | head -20
```

### 3.3 RabbitMQ 队列积压

**症状**：`workflow-task-queue` 或 `collaboration-task-queue` 持续高深度

**处理步骤**：
```bash
# 1. 查看队列深度
docker exec orin-rabbitmq rabbitmqctl list_queues -p / | grep -E "task|collab"

# 2. 检查 AI Engine worker 是否在消费
docker compose ps ai-engine
docker exec orin-rabbitmq rabbitmqctl list_consumers -p /

# 3. 如果 AI Engine 宕机导致积压，重启它
docker compose restart ai-engine

# 4. 手动触发失败任务重试（从管理界面或 API）
# POST /api/v1/tasks/{taskId}/retry
```

### 3.4 磁盘空间不足

**症状**：Docker 写入失败、日志文件巨大

**处理步骤**：
```bash
# 1. 定位大文件
du -sh */logs/* 2>/dev/null
docker system df

# 2. 清理 Docker 未使用资源
docker system prune -f --volumes

# 3. 清理旧日志（保留最近 7 天）
find orin-backend/logs -name "*.log.*" -mtime +7 -delete
find orin-backend/logs -name "*.json.*" -mtime +7 -delete

# 4. 清理 old Docker images
docker image prune -af
```

## 4. 生产变更 SOP

### 4.1 配置变更

1. **备份当前配置**：`bash scripts/backup.sh --keep 30`
2. **在测试环境验证**：修改 `.env` 或 `application.properties`，跑 `business-smoke.sh`
3. **申请变更窗口**：记录变更内容、预期影响、回滚计划
4. **执行变更**：在低峰期执行，优先改 `.env` 而非代码
5. **验证**：检查 `/actuator/health`、日志无 ERROR
6. **观察 30 分钟**：确认指标（错误率、延迟）无异常

### 4.2 版本升级

1. **确认回滚方案**：`scripts/rollback.sh` 已测试
2. **备份**：`bash scripts/backup.sh`
3. **拉取新代码**：`git pull`
4. **重建后端**：`cd orin-backend && mvn clean package -DskipTests`
5. **重建前端**：`cd orin-frontend && npm run build`
6. **重启服务**：`docker compose restart backend frontend ai-engine`
7. **验证**：跑 `business-smoke.sh`
8. **观察**：监控错误率、延迟、队列深度

## 5. 灾难恢复

### 5.1 数据库完全损坏

1. **停止所有服务**：`docker compose stop`
2. **确认备份可用**：`ls backups/orin-*/mysql/orindb.sql.gz`
3. **恢复数据库**：`bash scripts/restore.sh backups/orin-YYYYMMDD-HHMMSS --mysql`
4. **启动服务**：`docker compose up -d`
5. **验证**：
   - 后端健康：`curl http://localhost:8080/actuator/health`
   - AI Engine 健康：`curl http://localhost:8000/health`
   - Workflow 提交：跑 `business-smoke.sh`

### 5.2 所有容器数据丢失（volume 损坏）

1. **重建基础设施**：`docker compose down -v && docker compose up -d`
2. **确认 schema 初始化成功**：`docker logs orin-mysql 2>&1 | grep -E "V[0-9]+|Flyway"`
3. **恢复备份**：`bash scripts/restore.sh backups/orin-YYYYMMDD-HHMMSS --all`
4. **如无备份**：执行 `scripts/regenerate-schema-snapshot.sh` 重建 schema base，重新 Flyway 迁移

### 5.3 AI Engine 完全损坏

```bash
# 1. 停止后端（防止请求积压）
docker compose stop backend

# 2. 重建 AI Engine
cd orin-ai-engine
pip install -e .  # 或 docker compose build ai-engine

# 3. 重启 AI Engine
docker compose restart ai-engine

# 4. 验证
curl http://localhost:8000/health

# 5. 恢复后端
docker compose restart backend

# 6. 检查积压任务并重试
curl http://localhost:8080/api/v1/tasks?status=FAILED | jq '.tasks[] | .taskId'
```

## 6. 关键端口与日志路径

| 服务 | 端口 | 健康检查 | 日志路径 |
|------|------|----------|----------|
| 后端 | 8080 | `/actuator/health` | `orin-backend/logs/orin-backend.log` |
| 前端 | 5173 | `curl localhost:5173` | Docker container logs |
| AI Engine | 8000 | `/health` | Docker container logs |
| MySQL | 3306 | docker exec mysql ping | Docker volume |
| Redis | 6379 | `docker exec redis redis-cli ping` | Docker volume |
| RabbitMQ | 5672 | Management UI `:15672` | Docker volume |
