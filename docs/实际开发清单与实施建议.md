# ORIN 实际开发清单与实施建议

## 1. 文档目的

本文档基于当前 ORIN 的真实完成度，给出一份可直接落地的开发清单和实施建议，目标不是继续扩展“看起来很多的功能”，而是优先补齐核心闭环，提升平台可信交付能力。

## 2. 总体开发原则

### 2.1 先做真实闭环，再做新能力扩展

后续开发优先顺序应为：

1. 修正现状认知与文档
2. 稳定主链路
3. 补齐核心闭环
4. 再做高级能力和平台化能力

### 2.2 不建议回退到 Yuxi-Know 单体重做

当前仓库已经在 Spring Boot 上投入了大量主控层代码。
若强行回退到“FastAPI 单体 + Yuxi 原生代码复用”路线，成本会非常高，且会冲击现有模块。

建议正式确认当前路线：

- Java 后端继续作为系统主控层
- Python AI Engine 继续作为执行引擎
- Yuxi-Know 作为能力标杆和产品参考，而不是代码基座

### 2.3 定义“完成”的标准

一个模块只有同时满足下面条件，才应定义为“完成”：

- 有真实后端实现，不是 mock、placeholder、直接抛异常
- 有前端可用入口
- 有明确状态机和错误处理
- 有最小测试覆盖
- 有文档说明输入、输出、限制和运维方式

## 3. 实施阶段划分

## 3.1 阶段 A：真相对齐与基础稳定化

建议周期：`2-3 周`

目标：

- 让项目进入“可以可信开发和评审”的状态

交付标准：

- 所有主要文档与真实状态一致
- 测试和环境基线可用
- 未实现能力不再伪装为已完成

开发清单：

1. 文档口径统一
- 更新 README，删除或降级未闭环能力描述
- 更新系统评估文档，停止使用“90%+ 完成度”口径
- 为每个模块增加状态标签：`已上线`、`内测`、`开发中`、`占位`

2. 菜单与接口收敛
- 隐藏或标记知识图谱等占位入口
- 对未实现的控制器接口增加明确错误响应，不再直接抛 `UnsupportedOperationException`
- 为批量导入导出、刷新元数据、异步任务状态增加 feature flag 或禁用提示

3. 测试与环境修复
- 修复后端测试配置错误，移除 `application-test.properties` 中非法的 `spring.profiles.active`
- 明确 Python 引擎测试安装方式，补充 `pytest` 依赖准备说明
- 固化前端、后端、AI 引擎的本地开发启动步骤

4. 配置边界收敛
- 明确 dev、test、prod 三套环境变量来源
- 检查数据库、Milvus、Redis、外部模型接口的必填配置
- 给系统设置增加基础参数校验

实施建议：

- 这一阶段不要继续加新页面
- 重点是去掉“看似完成”的幻觉
- 这是后续所有阶段的前提

## 3.2 阶段 B：主链路闭环

建议周期：`4-6 周`

目标：

- 把知识库、工作流、智能体调用三条主链做成可真实运行、可追踪、可验收的闭环

交付标准：

- 主要功能可以从前端发起，到后端执行，到结果回收
- 失败时有明确错误原因
- 关键链路带统一 traceId

开发清单：

1. 智能体管理补完
- 真正实现批量导出
- 真正实现批量导入
- 真正实现刷新全部元数据
- 真正实现异步任务状态查询
- 为聊天、导入导出、刷新操作补审计记录

2. 工作流执行器补完
- 修复工作流代理的 provider 注入逻辑，禁止错误 fallback
- 补齐 `Iteration`、`Loop` 节点真实执行语义
- 把 `KnowledgeRetrieval` 从 mock 改成真实知识检索调用
- 明确每个节点的输入、输出、异常格式
- 统一 Java 后端与 Python 引擎的 DSL 协议版本

3. 知识库任务状态机
- 定义统一任务状态：`PENDING`、`RUNNING`、`SUCCESS`、`FAILED`、`RETRYING`
- 覆盖上传、解析、分块、向量化、同步、删除、重建
- 每一步都保存执行时间、失败原因、重试次数
- 提供任务详情接口和前端可视化状态

4. 统一链路追踪
- 所有执行型请求必须生成或透传 `traceId`
- 审计日志、工作流执行、知识库任务、智能体聊天统一关联 `traceId`
- 统一错误码与错误信息结构

实施建议：

- 这阶段最重要的是“执行可靠性”，不是“节点数量”
- 只保留真正能跑通的节点和链路
- 对难以短期补完的能力先降级，不要继续模拟

## 3.3 阶段 C：协作闭环与统一观测

建议周期：`4-6 周`

目标：

- 让 ORIN 具备真正的多智能体协作能力和清晰的观测主链路

交付标准：

- 至少支持串行、并行两种真实协作模式
- 协作请求可追踪到任务、子任务、上下文、结果和失败原因
- 系统级与应用级观测有统一入口

开发清单：

1. 多智能体协作最小闭环
- 定义统一协作数据结构：
  - `task`
  - `subtask`
  - `context`
  - `shared_memory`
  - `result`
  - `retry`
  - `rollback`
- 先补串行协作
- 再补并行协作
- 辩论/评审模式放到下一阶段

2. Python 协作节点去 mock
- `Delegate` 节点接真实 Agent/Workflow 调用
- `ParallelFork` 节点修复结果汇总
- `Consensus` 节点实现真实策略，而不是取第一个结果
- `Critic` 节点接入真实审查逻辑或 LLM 审查
- `MemoryRead/Write` 节点从上下文内存升级为共享存储接口

3. 统一观测体系
- 定义三层观测模型：
  - Prometheus：系统与基础设施
  - Langfuse：LLM/Agent 调用链
  - ORIN Audit/Trace：业务审计与入口追踪
- 后端明确哪些字段必须写入三层观测
- 前端只做统一入口和摘要，不重复造完整监控平台

4. 告警机制收敛
- 明确告警源：系统资源、调用失败、协作任务失败、同步失败
- 告警记录统一落库
- 邮件、钉钉、企微通知采用统一渠道抽象

实施建议：

- 不要在协作能力尚未成型时就急于引入复杂队列编排
- 先把单机/单进程协作语义做扎实，再扩展到异步分发

## 3.4 阶段 D：高级能力与平台化扩展

建议周期：`4-8 周`

目标：

- 将当前平台从“可用”提升为“有特色、可交付、可扩展”

开发清单：

1. 知识图谱真实接入
- 引入图谱构建任务流
- 定义实体、关系、图谱构建状态
- 若采用 Neo4j，需要明确存储模型与查询接口
- 前端从占位页升级为图谱列表、图谱详情、检索入口、可视化浏览

2. 多模态增强
- 实现图片 OCR 云服务接入或本地引擎治理
- 实现音频 ASR 云服务或本地 Whisper 接入
- 明确失败降级策略和资源消耗边界

3. 外部框架集成收口
- 定义统一外部接入模型：
  - 基本信息
  - 认证信息
  - 能力类型
  - 同步方向
  - 健康状态
  - 错误状态
- 对 Dify、RAGFlow 等统一用同一套接入抽象，不要各写各的状态逻辑

4. 端侧同步治理
- 增量同步机制
- checkpoint 机制
- 失败重放
- 冲突处理
- 权限校验
- 同步日志与状态页

5. 驾驶舱与运维工具
- 平台概览大盘
- 系统维护工具
- 配置中心增强
- 消息中心、帮助中心、统计分析等辅助模块升级为真实数据驱动

实施建议：

- 高级能力必须建立在阶段 B、C 的稳定基础上
- 若基础链路未稳，不应提前做“展示型功能”

## 4. 按模块拆解的实际开发建议

## 4.1 智能体管理模块

应该如何开发：

- 先补服务层未实现接口
- 再为控制器补统一返回结构和错误码
- 再补版本管理、导入导出、刷新等操作日志
- 最后再扩展批量化和异步处理能力

开发顺序：

1. 服务实现补全
2. 接口测试
3. 前端交互与状态反馈
4. 审计和追踪

## 4.2 知识库模块

应该如何开发：

- 先做任务状态机，不要先做图谱界面
- 先保证解析、分块、向量化、检索和失败处理可靠
- 再在此基础上做图谱、评估、多模态增强

开发顺序：

1. 任务状态统一
2. 解析链路补全
3. 检索链路改真实
4. 同步链路治理
5. 图谱能力接入

## 4.3 工作流模块

应该如何开发：

- 先收敛 DSL 模型和版本
- 先补执行语义，再扩编辑器体验
- 先保证“执行正确”，再丰富节点种类

开发顺序：

1. DSL 协议整理
2. 节点输入输出规范
3. 执行器去 mock
4. 错误传播与 trace
5. 可视化调试增强

## 4.4 协作模块

应该如何开发：

- 先定义最小协作协议
- 先做串行和并行
- 再扩展辩论、评审、监督者模式

开发顺序：

1. 数据模型统一
2. 共享状态机制
3. 串行协作
4. 并行协作
5. 失败重试和回退
6. 高级协作模式

## 4.5 监控与运维模块

应该如何开发：

- 不要同时自己造 Langfuse、Grafana、日志平台三套界面
- 只做统一入口、摘要卡片和与业务操作关联最强的部分

开发顺序：

1. 统一 traceId
2. 统一指标定义
3. 聚合视图
4. 告警规则和通知渠道
5. 运维工具与维护任务

## 5. P0 / P1 / P2 清单

## 5.1 P0：必须优先完成

- 修复文档与真实状态不一致问题
- 修复后端测试基线
- 隐藏或降级未实现菜单和接口
- 补齐智能体管理关键未实现服务
- 修复工作流代理 provider 注入逻辑
- 补齐知识库任务状态机
- 统一 traceId、错误码、审计结构

## 5.2 P1：形成可交付闭环

- 工作流执行器去 mock
- 协作模块先实现串行与并行真实执行
- 统一观测模型接入
- 告警规则落地
- 同步链路失败重放与状态治理

## 5.3 P2：强化平台竞争力

- 知识图谱真实接入
- 多模态解析增强
- 外部框架接入统一抽象
- 驾驶舱、统计分析、消息中心
- 高级协作模式

## 6. 推荐团队分工

- 主控后端负责人
  负责配置中心、权限、网关、审计、状态机、系统治理

- Python 执行层负责人
  负责工作流执行器、协作执行器、节点实现、执行 trace

- 前端负责人
  负责真实闭环页面，不再继续扩大 placeholder 面积

- 测试与交付负责人
  负责环境、自动化测试、回归、部署脚本和验收基线

## 7. 里程碑建议

建议按如下里程碑推进：

- 里程碑 1：文档与测试基线可信
- 里程碑 2：知识库、工作流、智能体主链路闭环
- 里程碑 3：协作与观测闭环
- 里程碑 4：图谱、多模态、端侧同步和外部集成增强

## 8. 最终建议

后续开发最重要的不是“继续增加功能名”，而是“降低伪完成比例”。

建议整个团队从现在开始统一采用以下原则：

- 页面存在不等于功能完成
- 接口存在不等于能力完成
- 能跑 demo 不等于可交付
- 有测试文件不等于测试体系可用

只要按照“先稳定、再闭环、后扩展”的顺序推进，ORIN 仍然有机会从当前的半成品平台，逐步演进为真正可交付的智能体管理中枢。
