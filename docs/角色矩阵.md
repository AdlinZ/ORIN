# ORIN 角色矩阵

本文档定义管理台首版角色化体验与后端权限收口口径。当前只使用已有角色，不新增数据库角色。

## 1. 角色定义

| 角色 | 定位 | 默认入口 | 管理台可见模块 |
|------|------|----------|----------------|
| `ROLE_SUPER_ADMIN` | 超级管理员 | `/dashboard/runtime/overview` | 全部管理模块 |
| `ROLE_PLATFORM_ADMIN` | 平台管理员 | `/dashboard/runtime/overview` | 运行监控、系统设置；组织权限默认不开启 |
| `ROLE_ADMIN` | 系统管理员 | `/dashboard/runtime/overview` | 全部管理模块 |
| `ROLE_OPERATOR` | 运维 / 执行人员 | `/dashboard/applications/agents` | 智能体、工作流、知识库 |
| `ROLE_USER` | 普通使用者 | `/portal` | 不进入管理台模块；可进入 `/portal/api-keys` 管理个人访问密钥 |

兼容口径：前端仍识别历史 `ADMIN` / `USER` 字符串；后端权限判断以 `ROLE_*` 为准。

## 2. 前端权限口径

- 菜单来源统一为 `orin-frontend/src/router/topMenuConfig.js`。
- 顶部菜单和侧边栏使用同一套 `roles` 声明过滤一级菜单与子菜单。
- 路由守卫同时检查父级模块和子页面 `meta.roles`，直接输入 URL 也必须满足角色要求。
- `/dashboard` 和监控总览会按角色默认入口跳转，避免普通用户或运维落到管理员视图。

## 3. 后端权限口径

- `/api/v1/users/**`、`/api/v1/departments/**`、`/api/v1/roles/**` 不再匿名放行。
- 用户、部门、角色管理控制器要求 `ROLE_ADMIN` 或 `ROLE_SUPER_ADMIN`；`ROLE_PLATFORM_ADMIN` 默认不具备组织权限。
- `/api/v1/users/profile/**`、`/api/v1/users/dashboard/**`、`/api/v1/users/avatar` 属于个人资料能力，仍走 JWT 认证，不作为管理员管理接口。
- `/api/v1/api-keys/**` 分为两类权限：`ROLE_ADMIN`、`ROLE_SUPER_ADMIN`、`ROLE_PLATFORM_ADMIN` 可做全局治理；`ROLE_OPERATOR`、`ROLE_USER` 只能创建、列表、启用、禁用、删除、轮换和查看自己拥有的 `CLIENT_ACCESS` Key 使用历史。
- 普通角色的 API Key 所有权只来自 JWT 当前用户，不能通过 `X-User-Id` 或 `targetUserId` 覆盖；访问他人 Key 按不可见处理，返回 `404`。
- API Key 明文回显、配额重置、`PROVIDER_CREDENTIAL` 与 `MCP_ENV` 密钥管理仍仅限管理员 / 平台管理员角色。
- `/v1/**` API Key 通道与 `/api/v1/**` JWT 业务通道继续隔离。

## 4. 后续边界

- 本阶段不新增审计员角色、不做租户隔离、不做资源级 ACL。
- 按 key 长期配额趋势、细粒度模型 / 知识库权限放入后续 Phase 2。
