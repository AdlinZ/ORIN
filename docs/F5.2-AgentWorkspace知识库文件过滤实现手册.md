# F5.2 AgentWorkspace 知识库文件过滤实现手册

## 1. 实现目标

在 `/dashboard/agents/workspace` 支持：

1. 每个知识库可选择一个或多个文档作为检索范围。
2. 发送消息时把“知识库 + 文档过滤”一起传到后端。
3. 检索层在向量检索与关键词检索路径都应用文档过滤。
4. （建议）会话级筛选配置可持久化，刷新后不丢。

## 2. 设计约定（建议）

### 2.1 前后端请求字段

消息请求体新增：

```json
{
  "message": "测试知识库",
  "kbIds": ["kb1", "kb2"],
  "kbDocFilters": {
    "kb1": ["docA", "docB"],
    "kb2": ["docX"]
  }
}
```

说明：

- `kbDocFilters` 的 key 是 `kbId`。
- value 为空数组或缺失表示该 `kbId` 不做文档过滤（即全量文档）。

## 3. 代码修改步骤

### 3.1 前端：workspace 发送过滤参数

文件：`orin-frontend/src/views/Agent/AgentWorkspace.vue`

1. 新增状态：`const kbDocFilters = ref({})`
2. 在“知识库”卡片里，为每个已选知识库展示文档多选列表（数据来自 `GET /api/v1/knowledge/{kbId}/documents`）。
3. `sendMessage()` 调用改为：

```js
await sendChatMessage(currentSessionId.value, {
  message: content,
  kbIds: attachedKbIds.value,
  kbDocFilters: kbDocFilters.value
});
```

文件：`orin-frontend/src/api/agent-chat.js`

- 无需新增接口，只需保证 `sendChatMessage` 透传 `data` 即可（当前已支持）。

### 3.2 后端：DTO 与 service 接口升级

文件：`orin-backend/src/main/java/com/adlin/orin/modules/conversation/dto/ChatMessageRequest.java`

新增字段：

```java
private Map<String, List<String>> kbDocFilters;
```

文件：`orin-backend/src/main/java/com/adlin/orin/modules/conversation/service/AgentChatService.java`

改造 `sendMessage` 中检索循环：

1. 取当前 `kbId` 对应 `docIds`：

```java
List<String> docIds = Optional.ofNullable(request.getKbDocFilters())
    .map(m -> m.get(kbId))
    .orElse(Collections.emptyList());
```

2. 调用支持过滤的新检索方法：

```java
retrievalService.hybridSearch(kbId, request.getMessage(), 5, null, null, null, null, docIds);
```

### 3.3 后端：检索层增加 docId 过滤

文件：`orin-backend/src/main/java/com/adlin/orin/modules/knowledge/service/RetrievalService.java`

1. 新增重载：

```java
public List<SearchResult> hybridSearch(..., List<String> docIds)
```

2. 在向量结果路径过滤：

- `childResults` 获取后，按 `metadata.doc_id` 过滤。
- 为空则直接返回空或降级关键词检索（按产品策略）。

3. 在关键词路径过滤：

- `keywordSearch(kbId, query, topK, docIds)`
- Repository 新增按 kbId + docIds + keyword 查询。

文件：`orin-backend/src/main/java/com/adlin/orin/modules/knowledge/repository/KnowledgeDocumentChunkRepository.java`

新增查询方法（示意）：

```java
@Query("SELECT c FROM KnowledgeDocumentChunk c WHERE c.documentId IN :docIds AND LOWER(c.content) LIKE LOWER(CONCAT('%', :keyword, '%'))")
List<KnowledgeDocumentChunk> searchByDocIdsAndKeyword(@Param("docIds") List<String> docIds, @Param("keyword") String keyword);
```

注意：`docIds` 为空时不要走这个查询，避免 SQL IN () 问题。

### 3.4 （建议）会话筛选配置持久化

当前 `AgentChatSession.attachedKbIds` 是 `@Transient`，不持久化。

建议新增迁移：

文件：`orin-backend/src/main/resources/db/migration/V53__Agent_Chat_Session_Filter_State.sql`

```sql
ALTER TABLE agent_chat_session
  ADD COLUMN attached_kb_ids_json TEXT NULL,
  ADD COLUMN kb_doc_filters_json TEXT NULL;
```

并在实体中改为持久化字段（JSON 字符串），service 里统一序列化/反序列化。

## 4. 验收标准

1. 仅选择文档 A 时，检索结果不包含文档 B 的内容。
2. 多知识库 + 各自文档过滤时，结果按各自过滤范围生效。
3. assistant 返回中 `retrievedChunks` 的 `source/doc_id` 与过滤条件一致。
4. 刷新页面后，会话筛选配置保持（若已实现持久化步骤）。

## 5. 建议的回归脚本（手工）

1. 在 kb1 上传 `docA(含关键词 alpha-only)`、`docB(不含 alpha-only)`。
2. workspace 仅选择 kb1 + docA，提问 `alpha-only`。
3. 预期：命中来源仅 docA。
4. 清空 doc 过滤再提问同样关键词。
5. 预期：可出现 docA/docB（按语义结果）。

## 6. 风险与回滚

1. 风险：过滤条件处理不当导致“检索为空”。
2. 风险：docId 过滤仅在向量路径生效，关键词路径遗漏，造成结果穿透。
3. 回滚：
   - 前端移除 `kbDocFilters` 传参。
   - 后端保留新字段但忽略该字段（兼容回滚）。
   - 若已加 `V53`，可保留字段不使用，避免回滚 DDL 风险。
