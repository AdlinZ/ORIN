# ORIN 统一开发总计划

> 版本：`2026-03-31`
> 目的：基于当前仓库真实实现、`docs/` 现有文档和原始设计方案，统一 ORIN 后续开发方向、实施顺序、验收口径与文档基线，输出一份可直接执行的“从头到尾”开发计划。

## 1. 这份文档解决什么问题

当前仓库同时存在三类信息：

- 原始设计蓝图：目标大、范围全，但技术路线和当前实现不完全一致
- 当前实现说明：部分文档已经开始按真实代码收口
- 历史乐观口径：少量文档仍把“有页面/有接口/有模块”直接写成“已完成”

这会导致三个问题：

- 计划容易建立在错误现状上
- 开发顺序容易先做展示型功能，后补闭环
- 验收时容易出现“文档说完成了，链路却跑不通”

因此从现在开始，ORIN 后续开发统一以本文档为总计划，以“现有 ORIN 架构上补齐闭环”为唯一实施路线。

## 3. 统一现状判断

## 3.1 当前真实架构

ORIN 当前真实架构不是原始设计中的“Yuxi-Know/FastAPI 单体”，而是：

- `orin-backend`：Spring Boot 主控后端
- `orin-frontend`：Vue 3 管理端
- `orin-ai-engine`：Python FastAPI 执行引擎

这意味着后续计划必须建立在双后端协同上，而不是假设 Yuxi-Know 原生能力已经自动具备。

## 3.2 当前模块成熟度判断

结合现有文档与代码抽样，建议统一按下列口径判断：

| 模块 | 当前判断 | 说明 |
| --- | --- | --- |
| 智能体管理 | 中高 | 主体链路存在，但服务能力、批处理、任务治理仍需逐项做实 |
| 知识库 | 中 | 上传、解析、检索有基础，图谱、多模态、同步治理仍未完全闭环 |
| 工作流 | 中 | 编辑能力强于执行能力，复杂节点语义和代理链路需补完 |
| 多智能体协作 | 中低 | 结构完整，但真实执行、汇总、回退和共享记忆成熟度不足 |
| 监控观测 | 中 | 面板较多，但 Langfuse、Prometheus、审计 Trace 尚未完全统一 |
| 系统管理与安全 | 中高 | 基础设施具备，但平台级治理、配置中心、权限统一语义仍需收敛 |
| 外部集成与端侧同步 | 中低 | 入口多、模型杂，统一接入和失败治理尚未收口 |
| 测试与发布可靠性 | 中低 | 有测试文件，不等于形成稳定测试基线和发布门禁 |

## 3.3 真实差距总结

相对原始设计，当前最大的差距不是“完全没做”，而是以下五类：

1. 主链路闭环不够扎实
2. 多模块状态机、错误码、追踪语义不统一
3. 多智能体协作和复杂工作流仍偏演示态
4. 观测、权限、配置、同步等治理能力没有真正平台化
5. 文档与代码口径仍未完全统一

## 4. 统一开发目标

后续开发目标分为三层：

### 4.1 第一层：可信开发

让 ORIN 成为一个“文档可信、环境可起、状态可追、问题可查”的平台，而不是“页面很多但真假难辨”的平台。

### 4.2 第二层：核心闭环

把以下主链路做成端到端可运行、可验收、可回归：

- 智能体接入与调用
- 知识库上传到检索
- 工作流定义到执行
- 协作任务创建到汇总
- Trace、审计、告警、任务状态回查

### 4.3 第三层：平台化扩展

在核心闭环稳定之后，再补齐：

- 知识图谱
- 多模态增强
- 外部框架统一接入
- 端侧同步治理
- 驾驶舱与运维工具

## 5. 明确不做什么

本轮总计划明确不做以下路线：

- 不回退到 Yuxi-Know 单体重构
- 不把 Java 主控后端整体改写成 FastAPI
- 不以“继续堆页面/菜单/占位接口”作为主要工作
- 不在基础链路没稳之前优先开发展示型高级功能

## 6. 统一实施原则

### 6.1 完成定义

一个能力只有同时满足以下条件，才算完成：

- 有真实后端实现，不是 mock、placeholder、默认抛异常
- 有前端可用入口或明确 API 调用入口
- 有状态机、错误处理和失败语义
- 有最小测试覆盖或 smoke 路径
- 有使用说明、限制说明和运维说明

### 6.2 优先级原则

统一采用：

- `P0`：影响主链路可用性和可信验收
- `P1`：影响模块闭环和平台一致性
- `P2`：提升体验、扩展能力和平台竞争力

### 6.3 架构原则

- Java 后端负责主控、治理、权限、任务、审计、配置
- Python AI Engine 负责 DSL/Agent/协作执行
- 前端负责统一入口、状态展示和操作收口
- 不新增平行实现，优先补完现有链路

### 6.4 数据与追踪原则

所有执行型能力统一要求：

- 统一 `traceId`
- 统一任务状态语义
- 统一错误码和错误结构
- 统一用户、资源、动作、结果审计字段

### 6.5 远端 Agent 无人值守原则

当任务由远端开发 agent 在无人值守时段执行（如夜间）时，统一采用以下原则：

- 除破坏性/安全/高成本不可逆决策外，不等待用户回复，按可逆默认方案持续推进
- 优先交付“可验证增量”，不要停留在分析结论
- 无法联调外部依赖时，先完成本地逻辑、降级语义、最小测试和阻塞记录
- 结果必须显式包含“自主假设、验证结果、剩余高风险待决事项”

## 7. 总体路线图

建议按 6 个阶段推进，总周期以 `20-28 周` 为基线；如果团队较小，可按相同顺序拉长周期。

### 阶段 0：文档收口与真相对齐

建议周期：`1-2 周`

目标：

- 建立唯一可信文档口径
- 把“历史目标”和“当前现状”明确分离

范围：

- 文档索引重整
- 模块状态标签统一
- 旧文档分级为“基线/参考/历史”
- README、docs 索引、验收类文档同步修正

交付物：

- 统一开发总计划
- 更新后的 `docs/README`
- 模块状态总表
- 历史文档清单与使用说明

退出标准：

- 团队评审时不再出现“旧设计等于当前实现”的说法
- 文档阅读顺序清晰
- 外部沟通时有统一口径

### 阶段 1：工程基线与环境稳定化

建议周期：`2-3 周`

目标：

- 让工程进入可信开发状态

核心任务：

- 修复本地开发、测试、脚本和环境配置基线
- 明确 dev/test/prod 配置来源
- 固化一键启动和最小 smoke 流程
- 对未闭环接口返回明确错误，而不是模糊失败
- 建立 feature flag 或模块状态标记

后端工作：

- 清理默认抛异常或不透明失败的服务接口
- 统一健康检查、配置检查、依赖缺失提示
- 补最小化启动诊断接口

AI 引擎工作：

- 固化依赖安装说明
- 明确 handler 可运行最小集合
- 输出节点支持矩阵

前端工作：

- 对占位页、半成品页增加明确标识
- 隐藏或降级不可用入口
- 把错误提示统一收口

测试工作：

- 固化前端/后端/AI 引擎三条 smoke 命令
- 建立 CI 最小门禁

退出标准：

- 新人按文档可在本地拉起最小链路
- 三端均有稳定 smoke 路径
- “能打开页面但实际不可用”的情况显著减少

### 阶段 2：核心主链路闭环

建议周期：`4-6 周`

目标：

- 把智能体、知识库、工作流三条主链做实

#### 2.1 智能体管理闭环

任务：

- 对齐智能体生命周期状态
- 补全批量导入、批量导出、刷新元数据、异步任务查询的真实实现
- 为接入、聊天、刷新、批处理补齐审计记录
- 建立失败回执与任务回查

验收标准：

- 前端可发起完整接入与管理操作
- 失败时能看到明确原因
- 管理动作能按任务和审计记录追溯

#### 2.2 知识库闭环

任务：

- 统一知识库任务状态机：上传、解析、分块、向量化、检索、删除、同步、重建
- 保存每一步耗时、失败原因、重试次数
- 统一文档处理与检索回路的 `traceId`
- 明确 OCR/ASR 未配置时的降级语义

验收标准：

- 文件上传到检索可端到端复现
- 任务失败能够定位在哪一步失败
- 文档、任务、向量、同步状态可回查

#### 2.3 工作流闭环

任务：

- 统一 Java 管理层与 Python 执行层的 DSL 协议版本
- 修复 provider 注入和执行代理的不确定行为
- 补齐 `iteration`、`loop`、`knowledge_retrieval` 等节点真实语义
- 为节点输入输出、异常结构、运行日志建立统一规范

验收标准：

- 至少一条典型工作流可稳定重复执行
- 复杂节点不再依赖 mock/placeholder
- 执行失败可直接定位到节点级别

退出标准：

- 智能体、知识库、工作流三条链路都具备可演示、可回归、可追踪能力

### 阶段 3：协作闭环与统一观测

建议周期：`4-6 周`

目标：

- 把多智能体协作从“架子在”做成“真能跑”
- 把观测从“多套工具并列”做成“统一入口”

#### 3.1 协作模型收敛

统一定义：

- `package`
- `task`
- `subtask`
- `shared_memory`
- `context`
- `result`
- `retry`
- `rollback`

#### 3.2 协作执行补完

任务：

- 优先做串行协作真实闭环
- 再做并行协作真实闭环
- 修复结果汇总、超时、取消、重试、回退
- `Delegate`、`ParallelFork`、`Consensus`、`Critic`、`MemoryRead/Write` 等节点接入真实逻辑

验收标准：

- 串行模式可稳定完成任务分解、执行、汇总
- 并行模式可正确汇总分支结果
- 协作任务可查看上下文、子任务、汇总结果和失败原因

#### 3.3 三层观测统一

统一模型：

- Prometheus：系统、基础设施、中间件
- Langfuse：LLM、Agent、工具、协作执行链
- ORIN Trace/Audit：业务入口、用户行为、资源动作、任务结果

任务：

- 统一字段标准
- 统一 trace 透传
- 前端建立统一观测入口
- 告警记录统一落库并统一通知渠道

验收标准：

- 任何一次执行都可以从业务入口追到模型调用和系统事件
- 告警来源、接收渠道、处理状态可追踪

### 阶段 4：知识图谱与多模态增强

建议周期：`4-6 周`

目标：

- 把设计方案里最有辨识度但目前不够扎实的能力做成真实闭环

#### 4.1 知识图谱

任务：

- 明确图谱构建任务流
- 定义实体、关系、图谱构建状态和版本
- 明确 Neo4j 存储模型与查询接口
- 前端升级为图谱列表、详情、构建状态、检索入口、可视化浏览

验收标准：

- 能从知识文档触发图谱构建
- 能查看图谱构建结果与失败原因
- 能从业务场景访问图谱查询结果

#### 4.2 多模态

任务：

- 图片 OCR 云服务或本地引擎治理
- 音频 ASR 云服务或本地引擎治理
- 多模态文件上传、解析、失败降级、资源限制说明
- 前端展示每种模态支持范围与处理状态

验收标准：

- 至少图片 OCR、音频 ASR 各形成一条真实链路
- 未配置外部能力时有清晰降级与提示

### 阶段 5：外部集成、端侧同步与平台治理

建议周期：`4-6 周`

目标：

- 把“入口很多”的集成能力收口成平台级治理模型

#### 5.1 外部框架统一接入

统一接入模型至少包含：

- 基本信息
- 认证信息
- 能力类型
- 支持动作
- 健康状态
- 最近同步状态
- 最近错误状态
- 速率与资源限制

任务：

- Dify、RAGFlow、n8n 等统一抽象
- 不同外部框架不再各自定义状态语义
- 建立健康检查与失败回放机制

#### 5.2 端侧同步治理

任务：

- 增量同步
- checkpoint
- 失败重放
- 冲突处理
- 权限校验
- 同步日志与回查界面

验收标准：

- 服务端到端侧、端侧到服务端至少各一条稳定同步链路
- 同步失败可重放、可定位、可审计

#### 5.3 平台治理能力

任务：

- 配置中心增强
- 驾驶舱数据化
- 消息中心、帮助中心、统计分析改为真实数据驱动
- 系统维护能力收口到统一操作台

### 阶段 6：发布准备与正式验收

建议周期：`2-3 周`

目标：

- 把平台从“开发完成”推进到“可稳定交付”

任务：

- 回归测试
- 压测与容量评估
- 安全检查
- 部署演练
- 回滚演练
- 演示脚本、验收脚本、运维手册、发布清单最终收口

退出标准：

- 发布/回滚可演练
- 核心能力有验收证据
- 形成对外版本说明和内部运维说明

## 9. 每阶段统一验收标准

每个阶段验收都必须至少覆盖以下六项：

1. 功能是否真实闭环
2. 接口和页面是否都可用
3. 错误处理是否清晰

5. 是否有最小测试或 smoke 证明

### 12.1 多智能体协作迁移任务（LangGraph + RabbitMQ + Redis）

为支持从当前 Java 主编排平滑迁移到“LangGraph 编排 + RabbitMQ 并发分发 + Redis 共享状态”，按两周拆解如下：

#### 第 1 周（并行协作 POC）

1. 保持现有对外 API 不变，新增协作执行引擎开关（`JAVA_NATIVE` / `LANGGRAPH_MQ`）
2. 仅对 `PARALLEL` 模式灰度启用 `LANGGRAPH_MQ`
3. 在后端执行器中增加 RabbitMQ 发布能力（消息体至少含 `packageId`、`subTaskId`、`traceId`、`attempt`）
4. 在 AI 引擎增加 MQ Worker，消费任务并回写结果
5. 建立 Redis 协作键规范：
- `collab:{packageId}:ctx`
- `collab:{packageId}:branch_counter`
- `collab:{packageId}:lock:{subTaskId}`
6. 增加并行 E2E 验收链路：创建包 -> 分解 -> 并行执行 -> 汇总 -> 完成

#### 第 2 周（可靠性与模式扩展）

1. RabbitMQ 增加重试、DLQ、TTL；消费端增加幂等键（`packageId:subTaskId:attempt`）
2. 统一 DB/Redis/MQ 三方状态语义（重试、超时、取消、人工接管）
3. 迁移 `SEQUENTIAL` 与 `CONSENSUS` 模式到 LangGraph 协作图
4. 统一 trace 贯穿：Java -> MQ -> Python -> Java 回写
5. 指标收口到现有协作指标接口（成功率、延迟、token、重试）
6. 完成灰度发布与回滚策略：
- 灰度：内部任务 -> 10% -> 50% -> 100%
- 回滚：一键切回 `JAVA_NATIVE`

#### 迁移约束

- 不改前端调用协议
- 不一次性删除现有 Java Orchestrator
- 先双轨运行，稳定后再收敛职责

## 13. 最终结论

ORIN 后续开发不应再围绕“还能加多少功能页面”展开，而应围绕“把当前已经承诺的主链路做成可信平台”展开。

统一判断是：

- 现阶段最重要的不是扩张，而是收口
- 最重要的交付物不是页面数量，而是闭环质量
- 最重要的技术路线不是回退重构，而是在现有双后端架构上完成平台化补完

从今天开始，ORIN 的开发、评审、排期、验收，统一按本文档执行。
